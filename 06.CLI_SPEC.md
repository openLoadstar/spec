# 💻 LOADSTAR CLI Specification

본 문서는 LOADSTAR CLI의 현행 명령어 체계를 정의합니다.

### 설계 원칙

LOADSTAR의 파일은 두 가지 접근 방식으로 나뉩니다:

| 영역 | 접근 주체 | 역할 |
|---|---|---|
| MAP/, WAYPOINT/ | AI 에이전트 / 개발자 **직접** 읽기·쓰기 | 작업 수행 도구, 협업자 간 소통 매체 |
| .clionly/ (LOG, TODO) | **CLI만** 읽기·쓰기 | 변경 이력, 작업 목록 등 일관성 유지가 필요한 데이터 |

- **CLI는 일관성 유지 도구이지, 모든 작업의 게이트웨이가 아닙니다.** AI는 WayPoint/Map을 자유롭게 편집하며, CLI는 그 변경을 추적(log)하고 검증(validate)하는 역할을 합니다.
- **요소 생성/삭제/편집은 AI가 직접 수행합니다.** 별도 CLI 명령 없이 md 파일을 생성·수정·삭제하고, 부모 요소의 WAYPOINTS/CHILDREN을 직접 갱신합니다.
- **Git 관리는 직접 수행합니다.** `git init`, `git commit`, `git push` 등을 직접 사용합니다.
- **메타 동기화는 환경별 메커니즘으로 처리합니다.** 레퍼런스 구현은 Claude Code 기준이며, 다른 AI 환경에서도 동등한 동기화 장치가 필요합니다. (상세: `08.META_SYNC.md`)

---

## 1. 초기화 (Init)

* `loadstar init`: 현재 디렉토리에 `.loadstar/` 폴더 구조를 초기화합니다.
    - 수행: `.loadstar/` 하위 전체 폴더(MAP, WAYPOINT, COMMON, .clionly/LOG, .clionly/TODO) 생성.
    - root Map 요소(`M://root`) 자동 생성.
    - Git 초기화는 포함하지 않음 — `git init`을 별도 실행.

---

## 2. 탐색 (Navigation)

* `loadstar show [FILTER]`: 전체 WayPoint 목록을 ADDRESS + STATUS 테이블로 출력합니다.
    - FILTER 미지정 시 전체 WayPoint 출력.
    - FILTER 지정 시 주소에 해당 키워드가 포함된 WayPoint만 필터링 (대소문자 무시).
    - 예시:
      ```
      loadstar show          # 전체 목록
      loadstar show cli      # 주소에 "cli" 포함하는 항목만
      loadstar show test     # 주소에 "test" 포함하는 항목만
      ```

---

## 3. TODO 관리 (Task Lifecycle)

WayPoint STATUS를 기반으로 자동 산출되는 작업 목록입니다. `todo sync`가 WayPoint 파일을 스캔하여 TODO_LIST를 갱신합니다.

- **저장 위치** (`.clionly/` 하위 — CLI 명령으로만 수정):
  - `.loadstar/.clionly/TODO/TODO_LIST.md` — sync가 관리하는 현재 작업 목록
  - `.loadstar/.clionly/TODO/WP_SNAPSHOT.json` — WP 파일 상태 캐시
- **직접 편집 금지**: `loadstar todo` 명령으로만 조작한다.
- **상세 규격**: `07.TODO_SPEC.md` 참조

* `loadstar todo sync`: 전체 WayPoint를 스캔하여 TODO_LIST를 동기화합니다.
    - MAP 순회 → WP 파일의 modTime/size를 캐시와 비교 → 변경분만 STATUS 재확인
    - S_IDL/S_PRG → 목록에 추가/갱신, S_STB → 목록에서 제거
    - REFERENCE 대상이 미완료면 `[BLOCKED]` 표시

* `loadstar todo sync [ADDRESS]`: 특정 WayPoint만 개별 동기화합니다.

* `loadstar todo list`: 현재 PENDING/ACTIVE/BLOCKED 목록을 출력합니다.

* `loadstar todo history`: 전체 WayPoint의 TECH_SPEC 완료 항목 `[x]`을 수집하여 최신순 출력합니다.

* `loadstar todo history [MAP_ADDRESS]`: 해당 Map 하위 WayPoint만 필터하여 히스토리를 출력합니다.

## 4. 로그 관리 (Log)

AI 또는 개발자가 `.loadstar/` 요소를 직접 수정한 경우 변경 코멘트를 CLI 로그로 남깁니다. 코드 수정 과정의 관찰 내용·결정 사항·이슈를 요소 단위로 기록합니다.

### 로그 KIND 정의

| KIND | 설명 |
| :--- | :--- |
| `NOTE` | 일반 관찰 또는 메모 |
| `DECISION` | 설계/구현 결정 사항 및 그 근거 |
| `ISSUE` | 발견된 문제, 제약, 버그 |
| `RESOLVED` | ISSUE 해결 내용 |
| `PROGRESS` | 구현 진행 상황 체크 |
| `MODIFIED` | AI 또는 개발자가 md를 직접 수정한 경우 변경 내용 코멘트 |

### 명령어

* `loadstar log [TIME_RANGE] [FILTER]`: 로그를 조회합니다.
    - TIME_RANGE: `Nd` (N일) 또는 `Nh` (N시간). 생략 시 전체 기간 조회.
    - FILTER: 주소, KIND, 내용에 대한 키워드 필터 (대소문자 무시). KIND와 정확히 일치하면 KIND 필터로 동작.
    - 최대 출력: 1000건, 최신순 정렬.
    - `--list`: 전체 요소 목록을 ID/ADDRESS/SUMMARY 테이블로 출력 (키워드 필터, 페이징 지원)
    - 예시:
      ```
      loadstar log                        # 전체 로그
      loadstar log 7d                     # 최근 7일
      loadstar log 3h                     # 최근 3시간
      loadstar log cmd_show               # 키워드 필터 (전체 기간)
      loadstar log 2d cmd_show            # 키워드 + 기간
      loadstar log ISSUE                  # KIND 필터
      loadstar log --list                 # 요소 목록 조회
      ```

* `loadstar log add [ADDRESS_OR_ID] [KIND] "[내용]"`: 지정한 요소에 대한 로그 항목을 기록합니다.
    - ADDRESS_OR_ID: 전체 주소 (`W://root/cli/cmd_log`) 또는 단축 ID (`cmd_log`)
    - KIND 생략 시 `NOTE`로 기본 설정
    - 수행: `.clionly/LOG/`에 타임스탬프와 함께 LOG 기록.
    - 예시:
      ```
      loadstar log add cmd_log "CODE_MAP 갱신 필요"                  # ID + NOTE (기본)
      loadstar log add cmd_log ISSUE "multiline 파싱 실패"           # ID + KIND
      loadstar log add W://root/cli/cmd_log NOTE "full address"      # 전체 주소
      ```

## 5. 질문 관리 (Question Management)

WayPoint의 `ISSUE.OPEN_QUESTIONS`를 전역 스캔하여 미결·보류 질문을 조회합니다. (상세 포맷: `02.SCHEMA_DEF.md §4`, `05.ELEMENT_FORMAT.md` Decision 섹션 참조)

질문 확인(CONFIRMED 전환)은 CLI 쓰기 명령 없이 **AI가 WP 파일을 직접 편집**하거나 **UI Questions 패널**을 통해 처리한다.

* `loadstar question`: OPEN + DEFERRED 질문 목록 출력 (기본).
    - 수행: 모든 `WAYPOINT/*.md` 파일에서 `[Q{N}]`, `[Q{N} DEFERRED]`, `[Q{N} CONFIRMED ...]` 라인을 추출.
    - 기본 출력: `ADDRESS`, `QID`, `STATE`, `REF`, `QUESTION` 테이블 (CONFIRMED 제외).
    - 예시:
      ```
      loadstar question

      ADDRESS                                Q    STATE     REF  QUESTION
      -------                                -    -----     ---  --------
      W://root/maintenance/orphan_delete     Q1   OPEN           MAP 삭제 시 산하 WP 처리 방식
      W://root/frontend/dashboard            Q3   DEFERRED       차트 라이브러리 선택
      ```

* `loadstar question [FILTER]`: 주소·키워드 부분일치 필터.

* `loadstar question --all`: CONFIRMED 항목까지 포함 출력.
    - 결정 이력 회고용. Git/DECISIONS 추적 없이 빠르게 훑을 때 사용.

* `loadstar question stats`: 전체 상태 집계 (OPEN / DEFERRED / CONFIRMED 수).

### 참고

- `CONFIRMED` 상태는 기본 출력에서 제외된다 (`--all`로 조회). 결정 이력은 Git과 `DECISIONS/`에 보존.
- Decision 파일 생성은 CLI에서 제공하지 않는다 (사람이 직접 작성하거나 UI를 통해 생성).
- Q 확인 방법: AI가 대화 중 질문이 해소되면 WP 파일의 Q 라인을 `[Q{N} CONFIRMED ...]`으로 직접 편집.

## 6. 검증 (Validation)

* `loadstar validate`: 모든 WayPoint / Data WayPoint / Map의 참조 링크와 부모-자식 정합성을 검증하여 구조적 문제를 검출합니다.
    - 검증 대상 주소: `W://` (WayPoint), `D://` (Data WayPoint), `M://` (Map)
    - 수행:
      1. WAYPOINT 파일 전체 스캔 → PARENT, CHILDREN, REFERENCE 주소 추출
      2. DATA_WAYPOINT 파일 전체 스캔 → PARENT, REFERENCE 주소 추출
      3. MAP 파일 전체 스캔 → WAYPOINTS 목록 추출 (W://, D://, M:// 혼재 가능)
      4. **전방 참조 검증**: 위에서 수집한 모든 참조 주소의 실제 파일 존재 여부 확인
      5. **역방향 정합성 검증**:
         - WP / dwp의 `PARENT`가 가리키는 Map의 `WAYPOINTS`에 본인 주소가 등록되어 있는지 확인
         - Map의 `WAYPOINTS`에 등록된 자식 요소의 `PARENT`가 본 Map을 가리키는지 확인
      6. OPEN_QUESTIONS의 `[Q{N} CONFIRMED <DECISION_ID>]` 에서 `DECISIONS/<DECISION_ID>.md` 파일 존재 여부 확인 (인라인 CONFIRMED는 DECISION_ID 없으므로 스킵)
    - 출력:
      - 문제 없으면: `"all references valid (N waypoints, K dwps, M maps checked)"`
      - 문제가 있으면: SOURCE, FIELD, ISSUE_TYPE(BROKEN / ORPHAN / DANGLING), TARGET 테이블 출력
        - `BROKEN`: 참조 주소의 파일이 존재하지 않음
        - `ORPHAN`: WP/dwp의 PARENT는 존재하지만 그 Map의 WAYPOINTS에 본인이 없음
        - `DANGLING`: Map.WAYPOINTS에 등록된 자식의 PARENT가 본 Map을 가리키지 않음
    - 예시:
      ```
      loadstar validate
      # all references valid (7 waypoints, 3 dwps, 2 maps checked)
      ```

## 7. 내부 운영 원칙 (Data Integrity)

* **Parent-Child Integrity**: 요소 생성/삭제 시 부모 요소의 `WAYPOINTS` 또는 `CONNECTIONS.CHILDREN` 섹션은 동기화되어야 합니다. AI가 직접 편집하는 경우 이 책임은 AI에게 있으며, `loadstar validate`로 사후 검증합니다.
* **History via Git**: 모든 변경 이력의 실제 데이터는 Git이 보관합니다. `.clionly/LOG`는 Git 탐색을 위한 인덱스 역할이며, 두 정보가 불일치할 경우 Git의 내용이 우선합니다.
* **Path-Based Storage**: 모든 메타데이터는 `03.ADDRESS_CONVENTION.md`에서 정의한 URI 경로를 기반으로 파일 시스템에 물리적으로 배치됩니다.
* **MODIFIED Log 권고**: AI 또는 개발자가 md를 직접 수정한 경우, `loadstar log [ADDRESS] MODIFIED "[코멘트]"`를 실행하여 변경 내역을 LOG에 기록할 것을 권고합니다.
* **메타 동기화**: 소스코드 수정 후 관련 WayPoint 갱신은 환경별 메커니즘(Hook, 규칙 파일)으로 리마인드하며, `loadstar validate`로 구조적 정합성을 검증합니다. (상세: `08.META_SYNC.md`)
