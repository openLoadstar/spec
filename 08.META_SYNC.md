# 🔍 LOADSTAR 메타 동기화 명세

본 문서는 코드 수정 시 `.loadstar/` 메타데이터 갱신 누락을 방지하는 전략을 정의합니다.

> **📢 레퍼런스 구현 범위 명시**
>
> 본 문서에 기술된 구체적 메커니즘(`CLAUDE.md` 자동 로드, `PostToolUse` Hook, 슬래시 커맨드 등)은 **Claude Code 환경 기준의 레퍼런스 구현**입니다.
>
> Cursor, Aider, Copilot, 직접 API 호출 등 다른 AI 환경에서도 **동등한 동기화 장치가 반드시 필요**합니다. 각 환경의 구체적 구현(예: `.cursorrules`, 시스템 프롬프트 템플릿, 사용자 스니펫 등)은 사용자가 환경에 맞춰 구축합니다. 커뮤니티 기여를 환영합니다.

---

## 1. 문제 정의

AI 에이전트가 코드를 수정할 때, 관련 WayPoint의 TECH_SPEC 체크박스나 STATUS가 함께 갱신되어야 하지만 누락되는 경우가 빈번하다.

### 누락 발생 원인
- 코드 수정에 집중하여 메타데이터 후처리를 잊음
- 세션이 종료되면서 후처리 작업이 중단됨
- 작업자가 매번 갱신 여부를 확인하는 것이 번거로움
- 여러 프로젝트를 오가며 작업 시 프로젝트별 규칙이 적용되지 않음

### 일관성 모델: 허용 범위 내 일관성 (Tolerable Consistency)

LOADSTAR는 **완전 일관성을 목표로 하지 않는다.** 이는 의도된 설계 결정이다.

- 매 코드 수정마다 모든 메타데이터를 갱신하는 비용(토큰, 시간)은 그 효과를 초과한다
- 사람 중심 개발에서도 코드와 문서의 완전한 동기화는 사실상 불가능하다
- AI 세션 단절로 인한 후처리 누락은 구조적으로 발생할 수밖에 없다

**목표 상태: "드리프트를 알고 있는 상태"**

## 2. 동기화 전략: 프롬프트 + CLAUDE.md + 사후 검증

### 2.1 핵심 원칙

메타 동기화의 가장 확실한 방법은 **AI의 컨텍스트에 규칙을 직접 주입하는 것**이다. 환경 자동화(Hook 등)보다 프롬프트/CLAUDE.md 기반 접근이 범용적이고 신뢰도가 높다.

| 계층 | 메커니즘 | 신뢰도 | 비고 |
|:---|:---|:---|:---|
| **세션 시작 프롬프트** | 사용자가 작업 요청 시 WayPoint 규칙을 명시 | 가장 높음 | 매 세션 적용 보장 |
| **CLAUDE.md** | 프로젝트/워크스페이스에 규칙 파일 배치 | 높음 | Claude Code가 자동 로드 |
| **Hook** | PostToolUse 편집 감지 리마인더 | 보조 | 프로젝트 디렉토리 종속 |
| **사후 검증** | `loadstar todo sync`, `loadstar validate` | 복구용 | 언제든 정합성 복구 |

### 2.2 세션 시작 프롬프트에 WayPoint 규칙 포함

매 세션 시작 시 사용하는 프롬프트에 다음 규칙을 포함한다:

```
WayPoint 작업 규칙 (이 세션 전체에 적용):
- 코드 수정 전: 대상 WayPoint TECH_SPEC에 [ ] 항목 등록, STATUS S_IDL→S_PRG
- 코드 수정 후: [x] YYYY-MM-DD로 체크, 전체 완료 시 STATUS→S_STB
- 항목 없이 코드 수정 금지. 빠른 버그 수정은 사후 등록 허용
- SUMMARY가 현재 기능과 다르면 갱신
```

이 규칙은 프롬프트에 포함되므로 환경이나 디렉토리에 무관하게 적용된다.

### 2.3 CLAUDE.md 배치

프로젝트별 또는 워크스페이스 상위 디렉토리에 CLAUDE.md를 배치하여, Claude Code 실행 시 자동 로드되도록 한다.

**워크스페이스 구조 예시**:
```
workspace/              ← CLAUDE.md (공통 WayPoint 규칙)
├── loadstar_cli/       ← CLAUDE.md (CLI 프로젝트 규칙)
├── loadstar_ui/        ← CLAUDE.md (UI 프로젝트 규칙)
└── loadstar_SPEC/
```

- 상위 디렉토리에서 여러 프로젝트를 오가며 작업 시: 상위 CLAUDE.md가 적용
- 개별 프로젝트에서 작업 시: 해당 프로젝트 CLAUDE.md가 적용
- `loadstar init` 실행 시 `.claude/` 디렉토리와 Hook 스크립트도 함께 생성 (보조 수단)

### 2.4 WayPoint 작업 흐름

```
AI가 작업 요청 수신
    ↓
대상 WayPoint 확인 → TECH_SPEC에 [ ] 항목 등록 → STATUS S_PRG
    ↓
코드 수정
    ↓
TECH_SPEC [x] YYYY-MM-DD 체크
    ↓
전체 완료 시 STATUS → S_STB, SUMMARY 갱신
    ↓
loadstar todo sync → TODO_LIST 자동 갱신
```

### 2.5 Hook (보조 수단)

Claude Code의 `PostToolUse` Hook은 **보조 리마인더**로 활용한다. 프로젝트 디렉토리에서 직접 Claude Code를 실행할 때만 동작하며, 핵심 동기화 수단이 아닌 추가 안전장치이다.

- `loadstar init` 시 `.claude/settings.json` + `hooks/loadstar-drift-check.sh` 자동 생성
- 소스코드 Edit/Write 시 TECH_SPEC 등록/갱신 리마인더 출력

### 2.6 todo sync 연동

`loadstar todo sync`는 WayPoint STATUS를 기반으로 TODO_LIST를 자동 산출한다:

- **S_IDL** WP → TODO에 PENDING으로 등록
- **S_PRG** WP → TODO에 ACTIVE로 등록
- **S_STB** WP → TODO에서 제거 (완료)
- REFERENCE 대상이 미완료면 `[BLOCKED]` 표시

AI가 별도로 `todo add/done`을 호출할 필요 없이, WayPoint STATUS만 관리하면 TODO가 자동 동기화된다. (상세: `07.TODO_SPEC.md`)

### 2.7 수동 검증 — loadstar validate

```
loadstar validate
```

모든 WayPoint와 Map의 참조 링크를 추적하여 깨진 참조를 검출한다.

## 3. 조정 세션 (Reconciliation Session)

드리프트가 누적되면 별도의 조정 세션을 통해 일괄 정합성을 맞춘다. 코드 구현 세션과 분리하여 운영한다.

**트리거 조건 (하나 이상 충족 시)**:
- SYNCED_AT 30일 초과 WayPoint가 존재하면서 관련 코드 변경 이력이 있는 경우
- `loadstar validate`에서 깨진 참조가 발견된 경우
- 사용자가 수동 조정 세션 요청

**진행 순서**:
1. `loadstar show`로 전체 WayPoint 상태 파악
2. `loadstar validate`로 깨진 참조 확인
3. SYNCED_AT 오래된 WayPoint 우선 갱신
4. WayPoint TODO 체크박스 상태 현행화
5. 완료 후 SYNCED_AT 날짜 갱신
