## 1. 배경 — 현재 LOADSTAR의 공백

LOADSTAR는 WayPoint를 통해 **WHAT(TECH_SPEC)** 과 **HOW(코드 + CODE_MAP)** 를 잘 잡는다. 그러나 느슨한 결합 원칙 아래 다음이 명시적으로 캡처되지 않는다.

| 분류 | 현재 캡처 | 누락 |
|:---|:---|:---|
| WHAT | TECH_SPEC, SUMMARY | — |
| HOW | 코드, CODE_MAP | — |
| **WHY** | (간접적, SUMMARY 일부) | 결정 근거 |
| **WHY-NOT** | 없음 | 거절된 대안 |
| **MUST/MUST-NOT** | 없음 | 불변조건, 비목표 |
| **TRADE-OFF** | 없음 | 수용한 단점 |

→ "기획 의도와 코드의 일치"를 검증하려면 굵은 글씨 4개가 필요하다. 그러나 모든 WP에 강제하면 LOADSTAR의 채택률(낮은 마찰)이 훼손된다.

---

## 2. 핵심 모델 — 삼각 검토 자산

**`업무(변경 이력) ─ 의도(INTENT) ─ 코드`** 가 한 WP에 묶인 상태를 "삼각 검토 자산"으로 정의한다.

비용/가치 구조:

| 단계 | 비용 | 가치 |
|:---|:---|:---|
| AI 검토 메시지 생성 | 저(가끔, WP당 1회) | 트리거 |
| 사용자 답변 | 중(인간 시간) | **진짜 자산** |
| 삼각 자산 누적 | 0 | 영구 |
| 이후 활용 | 선택 | 가변 |

**원칙**: 싼 트리거로 비싼 자산을 만들고, 활용은 필요할 때만 한다. 자원 투입 의사결정이 자산 누적과 분리된다.

---

## 3. 워크플로우

```
[INTENT 부실 WP 큐]                ← AI가 자동 식별
        ↓
[AI 가벼운 토론]                   ← 다관점 (Extractor / Skeptic / Historian)
        ↓
[검토 메시지 생성]                 ← 사용자에게 전달할 질문 형태
        ↓
[사용자 답변]                      ← 30분 내 처리 가능한 분량
        ↓
[INTENT 기록 → 삼각 자산 완성]
        ↓
[리소스 투여 여부 결정 (Lv1~Lv4)]
```

### 3.1 AI 토론의 역할 분담

단일 AI는 자기 가설에 갇힌다. 의도 추출은 다관점이 실제로 더 정확하다.

| 역할 | 질문 |
|:---|:---|
| **Extractor** | 코드가 무엇을 하는가? 의도는 무엇으로 보이는가? |
| **Skeptic** | 그 의도와 모순되는 코드 부분은 없는가? |
| **Historian** | git log / SPEC 변경 이력이 다른 의도를 시사하지 않는가? |
| **Synthesizer** | 셋의 입력으로 검토 메시지 초안 작성 |

### 3.2 검토 빈도

빈번할 필요 없음. 다음 트리거 중 하나면 충분:

- **STATUS 승격 시점** (S_PRG → S_STB) — WP별 자연스러운 게이트
- **정기 회고** (스프린트/월별) — 누락분 점진 청소
- **변경 빈도 높은 WP** — mtime 기반 자동 큐

## 4. 구현 — 가벼운 RAG 확장

### 4.1 RAG 난이도 평가

LOADSTAR는 이미 **WayPoint 단위 의미 청킹 + ADDRESS 체계**를 갖고 있다. 일반 코드베이스 대상 RAG보다 시작 난이도가 현저히 낮다.

**Tier 1 (★)** — 벡터 DB 불필요
- WP 메타데이터 + CODE_MAP.scope 기반 키워드/주소 검색
- LOADSTAR에는 이 정도로 충분할 가능성이 높음

**Tier 2 (★★★)** — 의미 검색 필요 시
- 코드 임베딩 + 벡터 DB + 하이브리드 + 리랭커
- "이 코드와 비슷한 패턴" 같은 질의가 생기면 추가

→ **Tier 1로 시작하고, 부족할 때 Tier 2 보강.**

### 4.2 토큰 예산 제한

```
[검토 요청]
   ↓
[Scope Resolver]    WP 주소 패턴(W://root/ui/*) + depth=1
   ↓
[Budget Guard]      입력 N토큰 / 출력 M토큰 hard limit
   ↓
[CLI Orchestrator]
```

- **공간 제한**: WP 주소 화이트리스트
- **깊이 제한**: 직접 코드는 full, 의존 WP는 SUMMARY만
- **토큰 예산**: 초과 시 abort + 사용자 확인
- **턴 제한**: 멀티에이전트 대화 max 3턴

### 4.3 CLI 연동

| 도구 | 헤드리스 모드 | MCP 클라이언트 | 비고 |
|:---|:---|:---|:---|
| Claude Code | `claude -p` | ✓ | Agent SDK도 가능 |
| Gemini CLI | `gemini -p` | ✓ | |
| Codex CLI | `codex exec` | 부분 | |

세 CLI 모두 비대화형 모드 + MCP 클라이언트 동작 가능. **`loadstar_mcp` 하나를 공유**하면 어느 CLI든 동일한 LOADSTAR 뷰를 받는다. Orchestrator가 직접 LLM API를 호출할 필요 없이 CLI를 subprocess로 호출하면 됨.

### 4.4 리소스 공유 원칙

- **읽기 공유, 쓰기 단일**: 모든 CLI는 read-only. 결과 기록은 orchestrator만.
- **세션 격리**: 같은 리소스를 봐도 각 CLI의 대화 컨텍스트는 분리.
- **스냅샷 고정**: 커밋 해시 기준으로 같은 시점을 본다 (race 방지).
- **사전 fetch + 분배**: 같은 컨텍스트를 N개 CLI가 각자 받으면 토큰 N배. orchestrator가 1회 fetch 후 분배.

## 5. 자원 투입의 단계적 결정

삼각 자산이 누적된 뒤, 그 위에 무엇을 얹을지는 독립적 의사결정.

| 레벨 | 기능 | 투입 시점 |
|:---|:---|:---|
| **Lv1** | 검색·열람만 (현재 LOADSTAR + INTENT) | 자산 도입 즉시 |
| **Lv2** | 변경 시 자동 정합성 체크 (코드 diff vs INTENT) | 자산 일정량 누적 후 |
| **Lv3** | AI 멀티에이전트 설계 검토 (본격 토론) | 정합성 가치 확인 후 |
| **Lv4** | INTENT 기반 자동 테스트 생성 | 상용화 단계 |

각 레벨이 **독립적 가치**를 가지므로, 강제 로드맵이 아니라 필요에 따른 선택 항목.

## 6. 방법론화 — LIP (LOADSTAR INTENT 보완 프로토콜)

```
1. WP 작성 시 INTENT는 선택. 막히면 비워두고 진행
2. STATUS S_STB 승격 시 INTENT 작성 권장 (강제 X)
3. 매 스프린트 종료 시 `loadstar review --missing-intent`
4. AI 어시스트로 검토 메시지 생성 → 작성자가 30분 내 답변
5. 누적 커버리지 추적, 60% 미만이면 다음 스프린트 우선순위
```

핵심: **"막히면 비우고 진행"** — 강제하지 않는 게 채택률을 결정한다.

## 7. 리스크 및 주의점

1. **검토 피로** — 매주 모든 WP 검토는 무리. 변경분 + 큐 상위 N개로 제한.
2. **AI 추정 의도의 위험** — 사람 승인 없이 자동 기록 금지. "초안" 명시.
3. **회고 의도의 함정** — 코드만 보고 추출하면 "현재 코드 = 의도"가 됨. 반드시 거절된 대안/제약도 묻기.
4. **커버리지 강박** — 모든 WP에 INTENT를 채우는 게 목표가 아니라, **채울 가치 있는 WP를 식별**하는 게 일. AI가 "이 WP에 INTENT가 가치 있는가" 판단도 포함.
5. **CLI별 출력 포맷 차이** — JSON 모드 강제 또는 파서 분리.
6. **stateless 호출 비용 누적** — 긴 토론은 외부 컨텍스트 저장 + 매 턴 필요분만 주입.
7. **방법론 vs 도구** — 명령어로 워크플로우를 박지 않으면 흐지부지됨. 방법론 + 도구 동시 출시.

---

## 8. 부수 효과

검토 세션 1회로 의도 외에도 함께 발견 가능:
- **WP-코드 불일치** (TECH_SPEC에 있는데 구현 안 된 항목)
- **SUMMARY 노화** (코드는 변했는데 SUMMARY 옛날 그대로)
- **DEPENDS_ON 누락** (코드상 의존하는데 메타에 없는 것)
- **암묵적 INVARIANT** (모두가 알지만 안 적힌 규칙)

→ ROI가 좋다. 검토 1회 = 의도 + 정합성 + 메타 신선도.
