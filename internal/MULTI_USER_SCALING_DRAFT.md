# 👥 LOADSTAR 다중 사용자 확장 검토

> **상태**: 검토 단계 (구현 미착수)
> **작성일**: 2026-05-06
> **전제**: 현 v0.1.0 SPEC은 1인 사용 기준으로 설계됨. 사람이 수동으로 작업 범위를 분할하면 소규모 협업(2~3인)까지는 무리 없이 동작한다. 본 문서는 **10인 규모 협업 시 발생할 문제와 대응 아키텍처 후보**를 정리한다.
> **현 시점 결론**: 즉시 구현하지 않는다. 실제 협업 수요가 생길 때 본 문서를 출발점으로 본격 설계에 들어간다.

---

## 1. 문제 인식

10인 정도의 다중 작업자가 같은 LOADSTAR 저장소를 공유할 경우, 다음 영역에서 문제가 누적된다.

### 1.1 동시성 / 머지 충돌

마크다운 기반 WP는 머지에 약하다.

- **WP 동시 편집**: TECH_SPEC 항목 추가, COMMENT 갱신이 같은 위치에서 발생 → git 머지 충돌
- **STATUS 전이 race**: 두 사람이 같은 WP를 다른 STATUS로 전이
- **체크박스 race**: 충돌은 없지만 상대가 한 일을 본인이 모름 → 작업 중복
- **Decision 파일 번호 충돌**: `wp_x.YYYY-MM-DD.001` 동시 생성 → 같은 파일명, push 거부 또는 한쪽 분실
- **Parent ↔ Child 원자성**: 자식 WP 생성과 부모의 CHILDREN 갱신이 두 파일에 걸침. 한쪽만 push되면 dangling

### 1.2 가시성 / 통보

git pull만이 정보 채널이라 늦다.

- 누가 어떤 WP를 작업 중인지 모름 (claim 없음)
- CODE_MAP scope 충돌 사전 감지 불가
- OPEN_QUESTIONS의 답변자(assignee) 명시 위치 없음
- 선행 WP 완료 알림 없음 — 매번 pull로 확인

### 1.3 권한 / 거버넌스

1인 환경의 "AI 직접 편집"이 그대로 통하지 않는다.

- S_STB 전이 권한 (자기 작업 자기 종료 시 부실 검증)
- OPEN_QUESTIONS의 결정권자 명시 필요
- Decision 파일 승인 절차
- Map 구조 변경 권한

### 1.4 AI 세션 격리

LOADSTAR가 AI 매개라는 점이 다중 사용자에서 특히 어렵다.

- 각자의 AI 세션이 격리됨 → A의 AI가 만든 컨텍스트가 B의 AI에 안 전해짐
- CONFIRMED 한 줄만 보고 B의 AI가 결정의 배경을 잘못 추론
- `LOADSTAR_INIT.md`가 휘발성 협업 메모로 진화해야 함
- 사용자별 `.claude/memory`는 동기화 안 됨 → 팀 컨벤션은 `CLAUDE.md`로 유지해야 함

### 1.5 `.clionly/` 모델 재검토

TODO_LIST·LOG는 "CLI 전용 + 로컬 캐시"로 설계됨.

- LOG: A가 남긴 흔적을 B가 보려면 commit/push 필요 → 본래 "변경 직후 흔적" 의미 약화
- TODO_LIST: 각자 sync 결과의 일관성 (mtime 캐시는 로컬)
- WP_SNAPSHOT.json: 로컬 캐시인데 머지 충돌 가능

## 2. 아키텍처 후보

### 2.1 Git-only (현행 유지)

- WP는 git, 별도 coordination 없음
- PR + 리뷰 + 자동 검증(CI)으로 흡수
- **장점**: 인프라 0, 오프라인 작업 자유
- **한계**: 충돌 빈도가 임계점 넘으면 머지 비용이 작업 비용을 압도. 10인 규모에서 한계 명확.

### 2.2 핫 패스만 서버, 콜드 패스는 git (하이브리드)

- 서버 매개: STATUS 전이, OPEN_QUESTIONS CONFIRMED, Decision 생성, claim
- git: TECH_SPEC 본문, COMMENT, IDENTITY
- **장점**: 위험한 전이만 게이트, 일상 편집은 자유
- **한계**: 두 진실 소스 동기화 책임. 서버 상태와 git 상태가 어긋날 수 있음.

### 2.3 단일 Worker 매개 (전면 서버 모델)

- AI가 `.loadstar/` 직접 편집 안 함 → 메시지 브로커를 통해 Worker에게 의도(intent) 전달 → Worker가 검증·적용·git commit
- **장점**: 충돌 원천 차단. claim, 의미 검증, Decision 번호 부여 등 단일 지점에서 처리.
- **한계**: Worker가 SPOF. 오프라인 작업 사실상 불가. AI 워크플로 변경 부담.

### 2.4 AI 기획자 + 결정적 코어 (supervisor-agent, 최종 수렴안)

10인 협업 가설 시 가장 유력한 모델로 합의됨.

**구조**:

- **결정적 코어** (DB/서버, 권장: loadstar_ui Spring Boot 백엔드 안)
  - 락, 트랜잭션, 스키마 검증, ID/번호 부여, parent-child 무결성
  - **LLM에 절대 안 맡김** — invariant 보장
- **AI 기획자(조율자) — 백그라운드 세션**
  - 중복 탐지, 작업 매핑, scope 제안, 충돌 중재
  - RAG 기반 (WP/SPEC/Decisions 임베딩)
  - 결정은 추천, 강제는 코어가
- **AI 업무자 — 각 개발자의 세션**
  - 조율자에게 lease(scope + TTL) 요청
  - 자기 lease 범위 내에서 자기 로컬 메타로 진행
  - `.loadstar/` commit/push 권한 없음 — 변경은 패치로 조율자에 제출
  - 불일치 발견 시 조율자에 보고 → 사람에게 통보

**라우팅**:
- 기계적 요청 (status 전이, 체크박스): 코어 직행 (fast / cheap / deterministic)
- 의미적 요청 ("이거 어디 속하나?", "이미 있나?"): 기획자 advisor (slow path, valuable)

**왜 이 형태**:
- 비결정성이 invariant를 깨지 못함 (코어가 막음)
- LLM 비용은 *진짜 가치 있는 곳에만* (중복 탐지, scope 제안)
- 단일 commit 권한으로 push race 제거
- lease 발급 시점에 동시성 문제의 다수가 사전 차단

## 3. 메시지 브로커 선택

10인 규모(분당 수~수십 이벤트)에 **Kafka는 과한 인프라**다.

후보:
- **NATS / Redis Streams**: 가볍고 충분, 단일 노드 운영 가능
- **Postgres LISTEN/NOTIFY**: DB 이미 사용 중이면 추가 인프라 0
- **gRPC 양방향 streaming**: 동기 confirm 필요할 때 단순
- **HTTP API + SSE 푸시**: 별도 메신저 없이 가장 운영 부담 적음

**권장**: `loadstar_ui` Spring Boot 백엔드에 합치고 SSE 또는 WebSocket으로 실시간 푸시. 별도 프로세스 안 띄움.

## 4. 합의된 운영 모델 (10인 가설)

### 4.1 역할 구분

- **업무 조율자 (Coordinator AI)**: 백그라운드 상시 세션 1개. `.loadstar/` 메타 관리만 담당. commit/push 권한 단독 보유.
- **업무자 (Worker AI)**: 각 개발자의 AI 세션. 조율자에게 lease 요청 후 작업.
- **개발자**: 불일치 보고 채널의 최종 수신자. 조율자가 자동 해결 못하는 충돌 발생 시 호출됨.

### 4.2 작업 흐름

1. 업무자가 조율자에게 작업 의도 표명 ("X 기능 추가")
2. 조율자가 RAG로 중복 검사 → scope 제안 (예: `M://root/cli` 산하 lease, TTL 30분)
3. 업무자가 lease 범위 내에서 자기 로컬 메타·코드 편집
4. 업무자가 변경분(패치)을 조율자에 제출 → 조율자가 SPEC 정합성·중복·scope 위반 검사 → commit
5. 불일치/이탈 발생 시 업무자가 조율자에 보고 → 사람에게 통보

### 4.3 접근 권한 제어 (선택)

- 조율자가 lease 발급 시 권한 레벨 적용 (예: 신규 멤버는 S_STB 전이 불가, Decision 생성 불가)
- 결정은 lease 메타에 기록, 코어가 강제

## 5. 프로토타입 들어가기 전에 굳혀야 할 결정

본격 구현 시점에 확정해야 할 항목들. 현재는 합의 전.

1. **Lease scope grammar**
   - 서브트리 단위 (광범위) vs 파일·연산 단위 (정밀)
   - 잠정 권장: 서브트리 read + 파일·연산 write

2. **변경 제출 형식**
   - 패치 기반 API (조율자가 검증 후 적용) vs 업무자 브랜치 push 후 머지
   - 잠정 권장: **패치 기반 API**. 머지 모델은 충돌을 조율자 쪽으로 다시 끌어옴.

3. **조율자 세션 복구**
   - 백그라운드 세션이 죽었을 때 lease·진행 컨텍스트 보존
   - 결정적 상태(lease, claim, assignee)는 외부 저장(Postgres/SQLite). LLM 컨텍스트는 RAG로 복원.
   - 안 그러면 매 재시작이 동기화 사고

4. **기획자 AI 형태**
   - (i) 사람이 띄우는 백그라운드 세션 (Claude Code 인스턴스 상시 가동)
   - (ii) 시스템이 자동 호출하는 stateless 함수 (Anthropic API 직접)
   - **합의**: (i) 백그라운드 세션 방향. 단, 결정적 상태는 (ii) 측 관점으로 외부화.

5. **오프라인 모드**
   - 허용하면 단일 Worker 모델 사실상 불가 → 하이브리드(2.2)로 후퇴
   - 허용 안 하면 4번 모델 그대로 진행

## 6. 추후 검토 예정 (구현 단계)

### 6.1 불일치 발생 시 대처

본 검토에서 가장 미해결 영역. 발생 가능 카테고리:

- **stale read**: 업무자가 본 WP가 그 사이 바뀜
- **scope 이탈 사후 감지**: 작업 중 lease 밖 영역을 손대야 함을 깨달음 → 추가 lease vs 롤백
- **cross-worker 의존**: A의 결과가 B의 가정을 깬 경우
- **외부 편집**: 개발자가 수동으로 `.loadstar/` 직접 편집 → 조율자 view와 git 실태 괴리
- **lease 만료 후 partial state**: 업무자 중도 이탈, 변경분 일부만 제출됨 → 회수 책임자

각 카테고리별 대응 절차(자동 vs 수동, 누구에게 통보)는 구현 단계에서 별도 정의.

### 6.2 RAG 신호 품질 검증

기획자의 "중복 탐지" 가치는 RAG 품질에 직결됨. 본격 구현 전 **최소 검증안**:

- 새 WP 생성 시 시스템이 "유사 WP 후보 N개"만 표시
- 결정은 사람이, 자동 권한 위임 0
- 이걸로 RAG 신호 품질이 충분한지부터 확인
- 충분 → 점진적 자동화 위임. 부족 → 기획자 AI 도입 자체 보류

### 6.3 소스코드 ↔ WP 원자성

AI가 코드 짜고 WP `[x]` 체크하는 게 두 시스템에 걸침. Worker(=조율자)가 "WP 변경 + 관련 git ref"를 한 트랜잭션으로 받는 형태로 처리하면 해결되지만 복잡도 ↑. 1차 구현에서는 *느슨한 결합* 허용 (각자 commit, 사후 정합성 검증).

## 7. 결론

- **현 v0.1.0 SPEC은 사람이 수동으로 작업 범위를 분할하면 소규모 협업까지 충분히 감당.** 즉시 구현 압박 없음.
- 10인 규모 시 **AI 기획자 + 결정적 코어** 모델이 가장 유력. `loadstar_ui` 백엔드 확장 형태로 구현하는 것이 자연스러움.
