# 📂 LOADSTAR Physical Storage Specification

본 문서는 LOADSTAR 시스템의 물리적 파일 저장 구조와 Git 연동을 통한 이력 관리 규약을 정의합니다. 논리적 계층(Map)과 상관없이 물리적으로는 평면적인 구조를 유지하여 탐색 효율성을 극대화합니다.

## 1. 디렉토리 구조 (Directory Structure)

모든 LOADSTAR 메타데이터는 프로젝트 루트의 `.loadstar/` 폴더 하위에서 유형별로 관리됩니다.

```
.loadstar/
├── MAP/            # M:// (Map 요소 저장소)
├── WAYPOINT/       # W:// (WayPoint 요소 저장소)
├── DECISIONS/      # 결정 기록 (OPEN_QUESTIONS → RESOLVED 의 결정 근거)
├── COMMON/         # 시스템 공통 파일 (설정 등)
└── .clionly/       # CLI 전용 접근 영역 (직접 편집 절대 금지)
    ├── LOG/        # md 요소별 변경 이력 인덱스
    ├── MONITOR/    # (예약)
    └── TODO/       # 현재 작업 목록 관리 (loadstar todo 명령으로만 수정)
        └── TODO_LIST.md    # 현재 PENDING/ACTIVE/BLOCKED 항목
```

> `.clionly/` 폴더는 `loadstar` CLI만 읽고 쓸 수 있습니다. AI 에이전트 및 개발자의 직접 편집은 엄격히 금지됩니다. 직접 편집 시 LOG와 실제 md 상태 간 정합성이 깨집니다.

## 2. 파일 명명 및 저장 규칙 (File Naming & Storage)

### 2.1 일반 요소 파일

모든 요소 파일은 논리 주소의 경로를 점(`.`)으로 연결한 이름으로 저장되며, 타입별 폴더 안에 평면적으로 배치됩니다.

* **명명 규칙**: `[TYPE_DIR]/[경로를_점으로_연결한_ID].md`
* **변환 규칙**: 논리 주소 `TYPE://경로/ID` → 파일 경로 `.loadstar/[TYPE_DIR]/경로.ID.md`
  - `W://root/cli/cmd_show` → `.loadstar/WAYPOINT/root.cli.cmd_show.md`
  - `M://root/cli` → `.loadstar/MAP/root.cli.md`
* **ID 유일성**: 동일 타입 폴더 내에서 변환된 파일명이 중복되지 않아야 합니다.
* **확장자**: 모든 메타데이터는 `.md` (Markdown) 형식을 사용합니다.

### 2.2 DECISIONS 파일 명명 규칙

`.loadstar/DECISIONS/` 하위는 날짜 기반 명명 규칙을 따릅니다 (주소 기반 아님).

* **명명 규칙**: `YYYY-MM-DD.slug.NNN.md`
  - `YYYY-MM-DD`: 결정 날짜 (시간순 정렬 자연)
  - `slug`: 주제 단어 (snake_case, 10~20자 권장)
  - `NNN`: 당일 순번 (001부터, 0 패딩)
  - 예: `2026-04-24.orphan_cascade.001.md`
* **참조 형식**: WayPoint 내 OPEN_QUESTIONS에서 `[Q{N} RESOLVED YYYY-MM-DD.slug.NNN]` 형식으로 역참조.
* **포맷**: `05.ELEMENT_FORMAT.md` Decision 섹션 참조.

### 2.3 .clionly 파일 명명 규칙

#### LOG 파일
`loadstar log` 명령 실행 시 CLI가 자동 생성합니다.

```
CL.[YYYYMMDD.HHMMSS].log.md
```

파일 내부 구조:
```
## LOG
- TARGET: W://root/cli/cmd_show
- KIND: NOTE
- AT: 2026-04-08T15:30:00
- CONTENT: 변경 내용 설명
```

## 3. Git 기반 이력 관리 (Git-Integrated Versioning)

LOADSTAR는 Git을 직접 사용하여 버전을 관리합니다.

### 3.1 물리적 이력 (Physical History)

* **Single Source of Truth**: 모든 `.loadstar/` 하위 파일(.clionly 포함)은 Git의 추적 대상입니다.
* **Git Responsibility**: 파일의 델타(차이점) 저장, 압축, 과거 시점 복구는 Git 엔진이 전담합니다.
* **`.clionly` 백업**: `.clionly/` 파일 역시 Git이 추적하므로, Git을 통한 복구가 가능합니다.

### 3.2 Git 커밋 관리

Git 커밋은 `git commit`을 직접 사용합니다. 커밋 시 `.loadstar/` 변경과 소스코드 변경을 함께 포함하는 것을 권장합니다.

### 3.3 메타 동기화 (Meta Synchronization)

소스코드 수정 후 관련 WayPoint 갱신 누락을 방지하기 위한 환경별 메커니즘:

* **Claude Code**: PostToolUse Hook 기반 리마인더 (`.claude/hooks/loadstar-drift-check.sh`)
* **IDE**: 규칙 파일 기반 (`.cursorrules` 등)
* **수동 검증**: `loadstar validate`로 깨진 참조 검출

> 상세: `08.META_SYNC.md`

## 4. 공통 시스템 파일 (COMMON Assets)

`COMMON/` 폴더에 저장되며 프로젝트 전체에서 공유되는 파일 목록입니다.

| 파일명 | 역할 |
| :--- | :--- |
| `git_config.json` | Git 연결 설정 (gitignored) |

## 5. 운영 규칙 (Operational Rules)

* **Parent-Child Integrity**: 계층 정보는 폴더 구조가 아닌 파일 내부의 `CONNECTIONS` 메타데이터를 통해 관리합니다. AI가 직접 편집하며, `loadstar validate`로 사후 검증합니다.
* **Direct Access**: 논리 주소를 해석하여 `://` 뒤의 경로를 점(`.`)으로 연결한 파일명으로 직접 타겟팅합니다.
* **CLI Exclusivity for .clionly**: `.clionly/` 내부 파일은 `loadstar` CLI만 생성·수정할 수 있습니다. 수동 편집 시 정합성 보장이 불가합니다.
* **LOG as Index**: `.clionly/LOG/`는 Git 탐색을 위한 인덱스 역할이며, 두 정보가 불일치할 경우 Git의 내용이 우선합니다.
