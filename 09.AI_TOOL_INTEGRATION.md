# 🔧 외부 AI 도구 통합 가이드

본 문서는 LOADSTAR 메타데이터를 외부 AI 도구(LSP, grep, MCP 서버 등)와 연동하여 **사람과 유사한 맥락 기반 코드 접근**을 구현하는 방법을 정의합니다.

> **위치**: 본 가이드는 SPEC의 **운용 권장**이며 강제 규약이 아닙니다. 환경별 도구 가용성에 따라 부분 적용 가능합니다.

---

## 1. 통합의 동기

### 사람이 코드를 다루는 방식

숙련된 개발자는 코드 작업 시 다음 흐름을 따른다.

1. **맥락 파악** — 어디 영역의 작업인지, 왜 하는지, 진행 상태는 어떤지
2. **재사용 가능성 점검** — 모델 클래스/DAO/유틸이 이미 있는지 함수명·시그니처로 확인
3. **수정 또는 신규 작성** — 적합한 게 있으면 재사용, 없으면 추가

이 흐름을 AI가 동등하게 수행하려면 **두 종류의 정보**가 필요하다.

| 정보 종류 | 출처 | 답하는 질문 |
|:---|:---|:---|
| **의도·맥락** (Why) | LOADSTAR (WayPoint, CODE_MAP) | 왜 이 작업을 하는가? 어디 영역인가? |
| **코드 사실** (What) | LSP, grep, AST 도구 | 이 함수가 어디 정의됐나? 누가 호출하나? |

LOADSTAR 단독으로는 의도만, LSP 단독으로는 사실만 다룰 수 있다. **두 정보를 결합**해야 사람과 유사한 맥락 접근이 가능하다.

## 2. CODE_MAP.scope — 의도와 코드의 연결점

`05.ELEMENT_FORMAT.md`에 정의된 `CODE_MAP.scope` 필드는 **WayPoint의 작업 의도와 실제 코드 위치를 잇는 다리** 역할을 한다.

```
### CODE_MAP
- scope: backend/src/main/java/com/loadstar/explorer/service/
- scope: frontend/src/features/waypoint-editor/
```

이 필드 하나가 외부 도구와의 통합 인터페이스다. AI는 다음 흐름으로 동작한다:

```
  ① WayPoint 로드 → CODE_MAP.scope 추출
       ↓
  ② scope를 외부 도구 호출 시 탐색 범위로 전달
       ↓
  ③ 외부 도구가 좁혀진 범위 내에서만 결과 반환
       ↓
  ④ AI가 결과를 가지고 재사용/생성 판단
```

**핵심 원칙**: 외부 도구 호출 시 scope 없이 전역 탐색하지 않는다. scope가 없는 WP는 작업 착수 전 작성한다 (`01.MASTER_GUIDE §5.2 Pre-Load` 참조).

## 3. LSP 도구 통합 (예: Serena)

### 3.1 Serena MCP란

**Serena**(https://github.com/oraios/serena)는 LSP(Language Server Protocol)를 MCP로 노출하는 서버다. AI가 IDE에서 사람이 사용하는 "정의로 이동", "참조 찾기", "심볼 목록" 같은 정확한 그래프 질의를 호출할 수 있다.

지원 언어: Java, Go, Python, TypeScript, Rust 등 LSP가 있는 모든 언어.

### 3.2 통합 흐름

```
사용자: "WayPoint editor의 저장 버튼 동작 수정해줘"
    ↓
AI: loadstar_show 또는 .loadstar/MAP 탐색 → 대상 WP 식별
    ↓
AI: WayPoint 파일 읽기 → CODE_MAP.scope 추출
    예: scope = ["frontend/src/features/waypoint-editor/"]
    ↓
AI: Serena 호출 시 scope를 인자로 전달
    serena.find_symbol(
      name="SaveButton",
      relative_path="frontend/src/features/waypoint-editor/"
    )
    ↓
Serena: 해당 디렉토리 내 심볼만 반환 (전역 탐색 회피)
    ↓
AI: 결과 분석 후 수정 또는 신규 함수 작성
```

### 3.3 LOADSTAR MCP가 없어도 동작

Claude Code 환경에서는 LOADSTAR MCP 없이 다음 조합으로 동등 동작 가능하다:

```
Read .loadstar/WAYPOINT/root.frontend.waypoint_editor.md
   → CODE_MAP.scope 파싱
   → Serena MCP 호출 시 scope 전달
```

외부 클라이언트(Claude Desktop, Cursor, Cline, 웹 Claude.ai 등)에서는 `Read`/`Bash`가 제한되므로 LOADSTAR MCP 서버가 필요하다 — `loadstar_mcp` 저장소에서 Python 1차 구현 진행 중. 상세는 §10 및 `internal/MCP_DESIGN_DRAFT.md` 참조.

## 4. grep / 검색 도구 통합

### 4.1 ripgrep, ag 등 직접 호출

CODE_MAP.scope를 path 인자로 그대로 전달:

```bash
# 좋음: scope 범위 내 검색
rg "saveWaypoint" frontend/src/features/waypoint-editor/

# 나쁨: 전역 검색
rg "saveWaypoint"
```

### 4.2 Claude Code 내장 Grep 도구

`path` 파라미터에 scope 전달:

```json
{
  "pattern": "saveWaypoint",
  "path": "frontend/src/features/waypoint-editor/"
}
```

### 4.3 Cursor / Continue 등의 검색

각 도구가 제공하는 `include`/`exclude` 또는 폴더 한정 옵션을 항상 활용한다.

## 5. 안티 패턴과 권장 패턴

### 5.1 명확한 안티 패턴

| ❌ 안티 패턴 | 문제 | ✅ 권장 |
|:---|:---|:---|
| "UserDao 클래스 어디 있어?" | WP 의도/상태 모르고 시작, 전역 검색 | "WP 확인 → scope 내 Serena 호출" |
| 패키지 파일 5개 일괄 Read로 구조 파악 | 토큰 폭발 | WP CODE_MAP → scope 내 ripgrep으로 클래스 목록 |
| WP-A 잠깐 → WP-B 잠깐 → WP-A 다시 | 컨텍스트 재로딩 비용 반복 | WP-A 작업 완전 종료 → WP-B 시작 |
| `paths` 없이 LSP 호출 | 인덱스 전체 스캔, 결과 노이즈 | scope 항상 동봉 |
| 코드부터 읽고 WP 사후 갱신 (기능 구현 시) | 의도 모르고 임의 결정 위험 | WP 먼저 (Pre-Load 표 참조) |

> **예외**: 수정 포인트가 명확한 단일 함수 버그·메시지 변경은 grep 직진이 더 효율적이다 (`01.MASTER_GUIDE §5.2 Pre-Load` 표 참조).

### 5.2 좋은 프롬프트 vs 나쁜 프롬프트

#### 나쁨 (전역 탐색 유도)
```
"login 관련 코드 좀 보여줘"
```
→ AI는 어디부터 봐야 할지 몰라 전체 grep, 결과 산만, 토큰 낭비.

#### 좋음 (WP 경유 명시)
```
"WayPoint W://root/backend/auth_service 확인 후
 해당 CODE_MAP.scope 안에서 login 관련 함수 보여줘"
```
→ AI가 WP를 먼저 로드 → scope = `backend/src/main/java/.../auth/` → 그 안에서만 탐색.

#### 좋음 (사용자가 작업 의도까지 전달)
```
"OAuth 추가 작업이야. 관련 WP의 CODE_MAP.scope에서
 기존 인증 처리 함수가 있는지 먼저 확인하고,
 없으면 신규 함수 위치를 제안해줘"
```
→ AI가 의도(OAuth 추가) + 영역(scope) + 행동(재사용 우선) 세 정보를 모두 받음.

### 5.3 워크플로우 강제 규칙 (CLAUDE.md에 권장)

```
코드 작업 시 다음 순서를 따른다:

1. 대상 WayPoint 확인 (loadstar show 또는 .loadstar/MAP/WAYPOINT 직독)
2. WayPoint의 CODE_MAP.scope를 후속 도구 호출 시 탐색 범위로 사용
3. scope 없는 WP에서 기능 구현 작업이면, scope를 먼저 작성한 후 진행
4. scope 내에서 기존 클래스/함수 재사용 가능 여부 점검
5. 수정 또는 신규 작성
6. TECH_SPEC 체크박스 갱신, 필요 시 STATUS 변경

전역 LSP/grep 호출(scope 없이)은 단일 함수 수정·메시지 변경 등
영향 범위가 명확한 경우에만 허용한다.
```

## 6. CLAUDE.md 권장 템플릿

워크스페이스 또는 프로젝트 루트에 다음을 배치한다.

```markdown
# {프로젝트명} — Claude Agent 규칙

## WayPoint 작업 규칙 (필수)

코드 수정 작업 시 반드시 아래 절차를 따른다.

### 작업 전
1. 대상 WayPoint 확인 (`.loadstar/WAYPOINT/` 하위)
2. TECH_SPEC에 작업 항목이 없으면 `- [ ] 작업 내용` 추가
3. STATUS가 `S_IDL`이면 `S_PRG`로 변경

### 코드 탐색 시
- WayPoint의 `CODE_MAP.scope`를 항상 탐색 범위로 사용한다
- LSP/grep 도구 호출 시 scope를 path 인자로 전달
- scope 없이 전역 탐색하지 않는다 (단일 함수 수정 등 명확한 케이스 제외)

### 작업 후
1. 완료된 TECH_SPEC 항목을 `- [x] YYYY-MM-DD 작업 내용`으로 체크
2. WP의 모든 TECH_SPEC이 완료되면 STATUS를 `S_STB`로 변경
3. SUMMARY가 현재 기능과 다르면 갱신
4. CODE_MAP scope가 변경되었으면 함께 갱신

### 원칙
- **항목 없이 코드 수정 금지** — 먼저 WayPoint에 "무엇을 할 것인가"를 기록
- 빠른 버그 수정은 코드 수정 후 사후 등록 허용
- 여러 WayPoint에 걸친 작업은 각각 갱신

## 외부 도구 사용 권장
- **LSP MCP** (예: Serena) — 심볼/참조 정확 탐색용. scope 전달 필수
- **검색 도구** (Grep 등) — path 인자에 scope 전달
- 자세한 통합 패턴은 `loadstar_SPEC/09.AI_TOOL_INTEGRATION.md` 참조
```

실제 사용 예시는 본 워크스페이스 루트의 `CLAUDE.md`를 참고할 수 있다.

## 10. LOADSTAR MCP 서버 (구현 진행 중)

### 10.1 위치

별도 저장소 `loadstar_mcp`에서 **Python**으로 1차 구현 진행 중 (2026-04-29~). WayPoint `W://root/mcp_server` 추적.

### 10.2 사용 환경 매트릭스

| 환경 | Read | Bash | LOADSTAR MCP |
|:---|:---:|:---:|:---:|
| Claude Code (CLI) | ✅ | ✅ | (있어도 됨) |
| Claude Desktop | ❌ | ❌ | ✅ 필요 |
| Cursor / Cline / Continue | 부분 | 부분 | ✅ 필요 |
| 웹 Claude.ai | ❌ | ❌ | ✅ 필요 |

→ **Claude Code 단독 사용자는 MCP 없이 §3의 직접 파일 읽기 + Bash 조합으로 충분.** MCP는 외부 클라이언트 이식성 도구.

### 10.3 노출 도구 (요약)

| 도구 | 동작 | 구현 |
|:--|:--|:--|
| `loadstar_show` / `loadstar_validate` / `loadstar_todo_list` / `loadstar_todo_history` / `loadstar_log` / `loadstar_log_add` / `loadstar_question` | CLI 명령 1:1 매핑 | CLI subprocess 래핑 |
| `loadstar_get_waypoint(address)` / `loadstar_get_map(address)` | 단일 요소 파싱 → JSON (CODE_MAP.scope 포함) | 직접 파일 읽기 |
| `loadstar_get_spec(section?)` | SPEC 본문 반환 (전체 또는 섹션) | 직접 파일 읽기 |

**의도적 제외**: `loadstar_check`, `loadstar_init`, `loadstar_todo_sync` (`internal/MCP_DESIGN_DRAFT §5` 참조).

### 10.4 설계 — Stateless

MCP 서버는 워크스페이스 상태를 보유하지 않는다. 모든 도구(`get_spec` 제외)는 `project_path` (절대경로)를 의무 인자로 받는다. 다중 프로젝트는 호출 시 다른 경로를 전달하면 자연 지원되며, 등록·발견 메커니즘은 노출하지 않는다 (`.loadstar/`가 곧 프로젝트 진실).

### 10.5 설치 (예정)

```
# Claude Code
claude mcp add loadstar -- uvx loadstar-mcp

# Claude Desktop / Cursor — claude_desktop_config.json
{
  "mcpServers": {
    "loadstar": {
      "command": "uvx",
      "args": ["loadstar-mcp"],
      "env": {
        "LOADSTAR_CLI_PATH": "/path/to/loadstar.exe",
        "LOADSTAR_SPEC_PATH": "/path/to/loadstar_SPEC"
      }
    }
  }
}
```

| 환경변수 | 필수 | 용도 |
|:--|:--|:--|
| `LOADSTAR_CLI_PATH` | 선택 | `loadstar` 바이너리 경로. 미설정 시 시스템 PATH 탐색 |
| `LOADSTAR_SPEC_PATH` | 선택 | `loadstar_get_spec` 호출 시에만 필요 |

**사용 패턴**: 사용자가 자기 프로젝트 절대경로를 알고 호출 시 함께 전달한다.
> 예: "C:/projects/myapp의 WP 상태 보여줘" → AI가 `loadstar_show(project_path="C:/projects/myapp")` 호출

### 10.6 Go 포팅 가능성

도구명·파라미터·JSON 스키마는 언어 중립적으로 설계. 외부 사용 검증 후 Go 포팅이 필요해지면 외부 클라이언트 영향 없이 교체 가능 (`internal/MCP_DESIGN_DRAFT §2.1` 참조).
