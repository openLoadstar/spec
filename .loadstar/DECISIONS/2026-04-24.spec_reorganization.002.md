### SOURCE
- WP: (해당 없음 — SPEC 수준 결정)
- Question: 공개 프로토타입 배포를 앞두고 SPEC 문서 번호·구조를 어떻게 정리할 것인가?

### CONTEXT
기존 SPEC은 13개 문서 번호가 비연속(06 중복, 09/10이 예시, 12가 내부용) 상태로 구성되어 공개 시 혼선 우려. 또한 측정 지표(12)는 유효성 검증용으로 비공개 유지가 필요하고, FLOW_CATALOG(09)는 내용이 거의 비어 있어 공개 시 미완성으로 비침.

### OPTIONS
1. **현상 유지** — 기존 번호 체계 그대로, 콘텐츠만 다듬기
2. **전면 재번호** — 8개 연속 번호 + examples/, internal/ 분리
3. **폴더만 분리** — 번호는 유지, examples/, internal/ 만 신설

### DECISION
**Option 2 (전면 재번호).**

- 01~08: SPEC 본문 (연속)
- `examples/`: WP_TEMPLATE, root_map, decision_sample
- `internal/`: EFFECTIVENESS_METRICS (gitignored, 로컬 전용)
- FLOW_CATALOG 제거

### RATIONALE
- 공개 시 독자는 번호 순서로 읽는다 — 비연속 번호는 "뭔가 빠졌나?" 의문 유발
- internal/ gitignore로 측정 지표는 유지 (유효성 검증 가능) 하면서 공개엔 노출 안 됨
- examples/는 이해를 돕지만 SPEC 본문과 성격이 달라 분리가 명확
- FLOW_CATALOG는 내용 부족 + 현재 미사용 → 프로토타입에 불필요

### IMPACT
- README 전면 재작성 (v1.3.11)
- 모든 교차 참조 새 번호로 갱신
- 파일명 변경: `01.LOADSTAR_MASTER_GUIDE → 01.MASTER_GUIDE`, `04.LOADSTAR_STORAGE → 04.STORAGE`, `08.LOADSTAR_CLI → 06.CLI_SPEC`, `11 → 07.TODO_SPEC`, `13.MONITOR_SPEC → 08.META_SYNC`
- git mv로 이력 보존

### COMMENT
공개 후 더 큰 재번호가 필요해지면 v2.0에서 진행. v1.x 범위는 이 번호 체계 유지.
</DECISION>
