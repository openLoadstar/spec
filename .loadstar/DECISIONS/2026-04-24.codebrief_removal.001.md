### SOURCE
- WP: (해당 없음 — SPEC 수준 결정)
- Question: AI가 코드를 보지 않고도 파일 구조를 파악하게 하려고 도입한 CodeBrief 개념을 유지할 것인가?

### CONTEXT
초기 SPEC에는 소스 파일 단위 구조 요약 문서인 CodeBrief(CB)가 정의되어 있었다. 의도는 AI의 코드 읽기 비용을 줄이는 것.

그러나 실제 운영 결과:
- 함수 시그니처 동기화 부담이 크다 (수동 갱신 의존)
- drift 감지 자동화 부재
- Claude Opus 4.7 급 모델은 1M 컨텍스트라 Read 비용이 낮아짐
- 생성 기준이 주관적 ("함수 10개 이상" 등)이라 과잉/과소 둘 다 발생

### OPTIONS
1. **유지 + 자동화 강화** — 함수 파싱 CLI로 CodeBrief 자동 생성·갱신
2. **제거** — CodeBrief 개념 자체를 SPEC에서 삭제
3. **축소** — 선택적 사용으로 격하, 필수 아님 표시

### DECISION
**Option 2 (제거).**

### RATIONALE
- 모델 성능 상승으로 CodeBrief의 핵심 가치(코드 탐색 비용 절감)가 희석됨
- 유지 비용이 효용을 초과
- "코드를 보지 않고 구조 파악"은 좋은 아이디어지만 현 단계에서 구체적 구현 아이디어가 부족
- 프로토타입은 개념 수를 줄이는 편이 학습 허들 낮춤

### IMPACT
- SPEC 8개 문서에서 CodeBrief 참조 제거
- `06.CODEBRIEF_TEMPLATE.md`, `06.BLACKBOX_TEMPLATE.md` 파일 삭제
- loadstar_ui의 `.loadstar/CODEBRIEF/` 레거시 폴더 정리
- v1.3.10 커밋에 반영

### COMMENT
향후 AI-friendly 구조 요약이 재도입될 수 있다. 그때는 **자동 생성 + 자동 drift 감지**가 선결 조건이어야 한다.
</DECISION>
