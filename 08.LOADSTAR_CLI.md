# 💻 LOADSTAR CLI Specification (v3.0)

본 문서는 요소의 생명주기 관리와 메타데이터 저장에 최적화된 LOADSTAR CLI 명령어를 정의합니다.

### 설계 원칙

LOADSTAR의 파일은 두 가지 접근 방식으로 나뉩니다:

| 영역 | 접근 주체 | 역할 |
|---|---|---|
| MAP/, WAYPOINT/, CODEBRIEF/ | AI 에이전트 / 개발자 **직접** 읽기·쓰기 | 작업 수행 도구, 협업자 간 소통 매체 |
| .clionly/ (LOG, TODO, MONITOR) | **CLI만** 읽기·쓰기 | 변경 이력, 작업 목록 등 일관성 유지가 필요한 데이터 |

- **CLI는 일관성 유지 도구이지, 모든 작업의 게이트웨이가 아닙니다.** AI는 WayPoint/BlackBox/Map을 자유롭게 편집하며, CLI는 그 변경을 추적(log)하고 커밋(checkpoint)하는 역할만 합니다.
- **CodeBrief는 필요 시 선택적으로 생성합니다.** 주요 소스 파일의 구조 요약이 필요할 때 `loadstar create CB` 명령으로 생성합니다.

## 1. 초기화 및 Git 연결 설정 (Init & Git Integration)

* `loadstar init`: 현재 디렉토리에 `.loadstar/` 폴더 구조를 초기화합니다.
    - 수행: `.loadstar/` 하위 전체 폴더(MAP, WAYPOINT, CODEBRIEF, COMMON, .clionly/LOG, .clionly/TODO) 생성.
    - root Map 요소(`M://root`) 자동 생성.
    - Git 저장소가 존재하면 자동으로 연결을 시도합니다.

* `loadstar git set [REPO_PATH_OR_URL]`: LOADSTAR가 연동할 Git 저장소를 설정합니다.
    - REPO_PATH_OR_URL: 로컬 경로 또는 원격 URL (예: `.`, `https://github.com/user/repo`)
    - 수행: `.loadstar/COMMON/` 에 Git 연결 정보를 저장합니다.

* `loadstar git status`: 현재 Git 연결 상태 및 연동 정보를 출력합니다.
    - 출력: 연결된 저장소 경로/URL, 현재 브랜치, 마지막 커밋 해시, 미커밋 변경 파일 수.

* `loadstar git unset`: Git 연결 설정을 해제합니다.

## 2. 요소 생명주기 관리 (CRUD)

모든 생성 명령은 부모 요소를 지정하여 LOADSTAR의 계층 구조(`M://` → `W://`)를 형성합니다.

* `loadstar create [TYPE] [ID] --parent [PARENT_ADDRESS]`: 새로운 요소를 생성합니다.
    - TYPE: `M`(Map), `W`(WayPoint), `CB`(CodeBrief) — 사람 또는 AI가 직접 생성하는 대상
    - ID: 프로젝트 전체 범위에서 고유해야 하며, CLI는 생성 시점에 중복 여부를 자동으로 검증
    - 수행: `ELEMENT_FORMAT.md`에 따른 기본 MD 파일 생성 및 부모 요소의 `WAYPOINTS`(Map) 또는 `CONNECTIONS.CHILDREN`(WayPoint) 항목에 자동 등록, LOG 기록

* `loadstar delete [ADDRESS]`: 지정된 요소를 삭제합니다.
    - 수행: 부모 요소의 `WAYPOINTS` 또는 `CONNECTIONS.CHILDREN` 리스트에서 해당 주소를 제거, LOG 기록

## 3. 체크포인트 (Checkpoint)

물리적 코드의 변화와 논리적 메타데이터의 변화를 하나의 시점으로 결합합니다.

* `loadstar checkpoint -m "[Message]" [--auto]`:
    1. `.loadstar/` 내 변경된 파일을 git staging.
    2. 커밋 메시지 자동 구성:
       - 사용자 메시지 + 변경된 `.loadstar/` 요소 주소 목록을 자동 append.
       - `--auto` 플래그 사용 시 `[AUTO-CHECKPOINT]` 접두사 부여 (자동 저장임을 명시).
    3. `git commit` 수행.
    4. 원격 저장소 설정 시 자동 push.
    5. `.clionly/MONITOR/checkpoint_needed.flag` 존재 시 삭제.

    커밋 메시지 예시:
    ```
    implement cmd_log

    변경 요소:
    - WAYPOINT/root.cli.cmd_log.md
    - MAP/root.cli.md
    ```

    자동 실행 시:
    ```
    [AUTO-CHECKPOINT] auto save

    변경 요소:
    - WAYPOINT/root.cli.cmd_create.md
    ```

### 3.1 자동 모니터링 (loadstar_monitor)

별도 프로세스 `loadstar_monitor.exe`가 `.loadstar/` 변경을 주기적으로 감시합니다.

* **스캔 주기**: 5분 간격 (기본값)
* **감시 대상**: `.loadstar/` 하위 요소 파일 (WAYPOINT/, CODEBRIEF/, MAP/)
* **동작**:
    1. `git status`로 `.loadstar/` 내 uncommitted 변경 확인.
    2. 변경 감지 시 `.clionly/MONITOR/checkpoint_needed.flag` 파일 생성.
       - 파일 내용: 변경 감지 시각 + 변경된 파일 목록.
    3. 이미 flag 파일이 존재하면 중복 생성하지 않음.
    4. `loadstar checkpoint` 실행 시 flag 파일 자동 삭제.
* **빌드**: `go build -o bin/loadstar_monitor.exe ./cmd/monitor`
* **실행**: `loadstar_monitor` (백그라운드 프로세스로 상주)

## 4. 탐색 (Navigation)

* `loadstar show [ADDRESS] [--depth N]`: 특정 요소의 메타데이터 내용을 터미널에 출력합니다.
    - `--depth N`: N 단계 하위 요소까지 트리 형태로 출력 (기본값: 0, 현재 요소만)

## 5. TODO 관리 (Task Lifecycle)

`TODO_LIST`를 통해 현재 해야 할 작업만을 관리합니다.

- **저장 위치** (`.clionly/` 하위 — CLI 명령으로만 수정):
  - `.loadstar/.clionly/TODO/TODO_LIST.md` — 현재 PENDING/ACTIVE/BLOCKED 항목만 보관
- **직접 편집 금지**: 테이블 데이터는 반드시 아래 명령으로만 조작한다. 직접 편집 시 `|` 구분 파싱이 깨질 수 있다.
- **완료 이력**: 완료된 작업의 이력은 WayPoint TECH_SPEC의 `[x] YYYY-MM-DD` 항목으로 추적한다. CLI TODO에는 별도 이력을 보관하지 않는다.

* `loadstar todo add [ADDRESS] "[Summary]" [--depends [ADDRESS]]`: TODO_LIST에 새 TODO 항목을 추가합니다.
    - ADDRESS: 작업 대상 `W://` WayPoint 주소
    - 수행: `.loadstar/.clionly/TODO/TODO_LIST.md` 테이블에 행 추가 (상태: `PENDING`, 발생 시간 자동 기입)
    - `--depends`: 선행 완료 조건인 Address. 미입력 시 `-`로 저장.
    - 저장 형식: `| W://address | YYYY-MM-DD HH:MM | Summary | PENDING | Depends_On |`

* `loadstar todo update [ADDRESS] [STATUS]`: TODO 항목의 상태를 변경합니다.
    - 허용 상태: `PENDING`, `ACTIVE`
    - `BLOCKED`는 `Depends_On` 기반으로 자동 계산되며 수동 설정 불가.
    - `COMPLETED` / `FAILED`는 이 명령으로 설정 불가 — `loadstar todo done`을 사용한다.
    - 수행: Address 완전 일치로 행 탐색 → Status 컬럼만 교체, 나머지 컬럼 보존.

* `loadstar todo done [ADDRESS]`: 해당 Address의 TODO 항목을 완료 처리합니다.
    - 수행: `TODO_LIST`에서 해당 행 삭제.
    - 부수 효과: 완료된 항목을 `Depends_On`으로 참조하는 다른 항목의 상태를 재평가.

* `loadstar todo delete [ADDRESS]`: TODO 항목을 취소/삭제합니다.
    - 수행: `TODO_LIST`에서 해당 행 삭제.

* `loadstar todo list`: 현재 PENDING/ACTIVE 상태의 TODO 목록을 출력합니다.
    - `Depends_On`에 미완료 항목이 있는 행은 Status를 `[BLOCKED]`로 표시하여 출력.

## 6. 로그 관리 (Log)

AI 또는 개발자가 `.loadstar/` 요소를 직접 수정한 경우 변경 코멘트를 CLI 로그로 남깁니다. 코드 수정 과정의 관찰 내용·결정 사항·이슈를 요소 단위로 기록합니다.

### 로그 KIND 정의

| KIND | 설명 |
| :--- | :--- |
| `NOTE` | 일반 관찰 또는 메모 |
| `DECISION` | 설계/구현 결정 사항 및 그 근거 |
| `ISSUE` | 발견된 문제, 제약, 버그 |
| `RESOLVED` | ISSUE 해결 내용 |
| `PROGRESS` | 구현 진행 상황 체크 |
| `MODIFIED` | AI 또는 개발자가 md를 직접 수정한 경우 변경 내용 코멘트 |

### 명령어

* `loadstar log [ADDRESS] [KIND] "[내용]"`: 지정한 요소의 BlackBox에 로그 항목을 추가합니다.
    - ADDRESS: 로그를 남길 요소의 논리 주소 (예: `W://root/cli/cmd_create`)
    - KIND: 위 표에서 정의한 로그 종류
    - 수행: `.clionly/LOG/`에 타임스탬프와 함께 LOG 기록.
    - 저장 형식:
      ```
      - [2026-03-06T15:30:00] [KIND] 내용
      ```
    - `MODIFIED` 사용 예시 (md 직접 수정 후):
      ```
      loadstar log W://root/cli/cmd_create MODIFIED "TECH_SPEC 1번 항목 완료 처리, STATUS S_IDL → S_PRG 변경"
      ```

* `loadstar findlog [OFFSET] [LIMIT] [--address ADDRESS] [--kind KIND]`: 로그를 조회합니다.
    - OFFSET: 건너뛸 항목 수 (0 = 가장 최근부터)
    - LIMIT: 출력할 최대 항목 수
    - `--address ADDRESS`: 특정 요소의 로그만 필터링
    - `--kind KIND`: 특정 KIND만 필터링
    - 미옵션 시 전체 로그를 최신순으로 출력.

    예시:
    ```
    loadstar findlog 0 10                                          # 전체 최근 10건
    loadstar findlog 0 20 --address W://root/cli/cmd_create        # 특정 요소 최근 20건
    loadstar findlog 0 5 --kind ISSUE                              # 전체 ISSUE 최근 5건
    loadstar findlog 0 10 --kind MODIFIED                          # 직접 수정 이력 최근 10건
    loadstar findlog 10 10                                         # 11번째부터 10건 (페이징)
    ```

## 7. 동기화 (Sync) — 추후 구현

* `loadstar sync`: Git log를 분석하여 누락된 LOG를 소급 생성합니다.
    - 대상: 개발자가 `loadstar checkpoint`를 거치지 않고 직접 `git commit`한 경우, 또는 AI가 md를 직접 수정하고 `loadstar log`를 남기지 않은 경우.
    - Git 연결 필수.

---

## 8. 내부 운영 원칙 (Data Integrity)

* **Parent-Child Integrity**: 요소 생성/삭제 시 부모 요소의 `WAYPOINTS` 또는 `CONNECTIONS.CHILDREN` 섹션은 실시간으로 동기화되어야 합니다.
* **History via Git**: 모든 변경 이력의 실제 데이터는 Git이 보관합니다. `.clionly/LOG`는 Git 탐색을 위한 인덱스 역할이며, 두 정보가 불일치할 경우 Git의 내용이 우선합니다.
* **Path-Based Storage**: 모든 메타데이터는 `ADDRESS_CONVENTION.md`에서 정의한 URI 경로를 기반으로 파일 시스템에 물리적으로 배치됩니다.
* **MODIFIED Log 권고**: AI 또는 개발자가 md를 직접 수정한 경우, 반드시 `loadstar log [ADDRESS] MODIFIED "[코멘트]"`를 실행하여 변경 내역을 LOG에 기록할 것을 권고합니다.
