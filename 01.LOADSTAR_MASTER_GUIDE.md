# 📖 LOADSTAR System Master Guide for AI Agents (v3.0)

이 문서는 LOADSTAR의 논리 구조와 운영 규약을 정의합니다. AI 에이전트는 모든 작업 시작 전 이 가이드에 정의된 논리적 관계와 물리적 접점의 구분 원칙을 숙지해야 합니다.

## 1. 시스템 철학: 논리(Logical)와 물리(Physical)의 분리
LOADSTAR는 복잡하게 얽힌 설계 의도와 작업 간의 관계를 **Link**로 엮고, 실제 구현된 물리적 결과물은 **SavePoint**로 고정합니다. AI는 '왜 이 작업을 하는가'를 알기 위해 Link를 탐색하고, '무엇을 고쳐야 하는가'를 알기 위해 SavePoint를 호출합니다.

### 1.1 일관성 원칙: 허용 범위 내 일관성 (Tolerable Consistency)

LOADSTAR는 코드와 메타데이터의 **완전 일관성을 목표로 하지 않는다.** 이는 의도된 설계 결정이다.

- 매 코드 수정마다 모든 메타데이터를 갱신하는 비용(토큰, 시간)은 그 효과를 초과한다
- 사람 중심 개발에서도 코드와 문서의 완전한 동기화는 사실상 불가능하다
- AI 세션 단절로 인한 후처리 누락은 구조적으로 발생할 수밖에 없다

**목표 상태: "드리프트를 알고 있는 상태"**
- 일상 작업: 가능한 범위에서 메타데이터를 갱신하고, 모니터링이 드리프트를 기록한다
- 주기적 조정: MONITOR/DIFF_LOG를 기반으로 AI 조정 세션에서 누적 드리프트를 선택적으로 해소한다
- AI는 메타데이터가 완벽하지 않을 수 있음을 전제하고, SYNCED_AT과 MONITOR/DIFF_LOG를 교차 확인하여 신뢰도를 판단한다

> 세부 조정 세션 운영 규칙은 `MONITOR_SPEC.md §8` 참조.

## 2. 핵심 요소 (Core Elements)
1. **Map (M)**: WayPoint의 집합을 구조적으로 묶는 길잡이 컨테이너. 프로젝트 계층을 표현하고 WayPoint 탐색 경로를 제공하는 데만 사용한다. **작업의 요청·실행·검증 주체가 되지 않는다.** 작업에 관한 모든 의도와 책임은 WayPoint가 담당한다.
2. **WayPoint (W)**: 모든 작업의 실행 단위이자 요청 주체 (의도의 최소 노드). 구현, 테스트, 문서화 등 모든 작업은 WayPoint 단위로 정의되고 추적된다. 하위 WayPoint를 가질 수 있다. 코드 수정이 수반되는 경우 CODE_MAP 섹션을 통해 탐색 범위를 지정한다.
3. **History (H)**: 모든 요소의 시간축 기반 변경 이력.
4. **Link (L)**: **[논리적 관계도]** 요소 간의 참조(분석/테스트), 선후(Predecessor/Successor), 의존 관계를 정의하는 지능형 연결망.
5. **SavePoint (S)**: **[물리적 접점 목록]** Git 커밋 해시, 파일 경로 및 라인 범위, 리소스 URL 등 실제 데이터의 위치 정보를 담은 인덱스.

### 2.1 WayPoint CODE_MAP
- **CODE_MAP**: WayPoint 내 섹션. 작업에서 수정해야 할 코드의 **탐색 범위(scope)**를 기술한다. 디렉토리 경로 단위로 범위를 한정하여 AI의 grep/탐색 비용을 절감한다.
- 파일 단위 구조 요약은 현재 SPEC에서 제공하지 않는다. 코드 구조 파악은 CODE_MAP scope 범위 내에서 grep/Read로 수행한다.

## 3. 주소 체계 (Addressing Convention)
- **형식**: `[요소약어]://[부모경로]/[고유ID]`
- **예시**: 
  - `L://root/dev/auth_to_test` (개발 작업과 테스트 케이스 간의 논리적 참조 링크)
  - `S://root/dev/auth_logic/git_v1` (특정 코드 형상의 물리적 저장 위치)

## 4. 데이터 규약: 계층적 참조 (Unified Schema)
모든 요소는 `ELEMENT_FORMAT.md` 규격을 준수하며, 관계와 자원을 다음과 같이 엄격히 구분하여 기록합니다.

### A. CONNECTIONS (논리적 관계 - Link 중심)
- **참조(Reference)**: 관련 분석 문서, 지식 베이스, 타 모듈과의 지식적 연결.
- **선후(Sequential)**: 작업의 전후 순서 및 작업 간 의존성 관리.
- **검증(Test)**: 해당 작업 결과물을 검증하는 테스트 케이스 노드와의 연결.

### B. RESOURCES (물리적 위치 - SavePoint 중심)
- **Git 접점**: Git Commit Hash, Branch, Tag 정보 등 형상 관리 시스템과의 연결.
- **파일 좌표**: 실제 소스 코드의 물리적 경로 및 수정된 구체적인 라인 범위.
- **리소스 위치**: 외부 문서 URL, 바이너리 저장소 경로, 설계 자산(Figma 등)의 위치 정보.

## 5. AI 행동 강령 (Operational Protocol)
1. **Understand Intent**: Map과 WayPoint를 통해 현재 작업의 목적을 먼저 파악한다.
2. **Trace Logic (Link)**: `L://` 주소를 추적하여 선행 작업의 맥락이나 참조해야 할 분석/테스트 정보를 확보한다.
3. **Target Physicals (SavePoint)**: 실제 작업 수행 직전, `S://` 주소에 접근하여 수정할 파일의 경로와 Git 위치를 확보한다.
4. **Execute & Record**: 작업을 수행하고 물리적 결과(변경 파일, Git Hash)는 새로운 `SavePoint`에, 논리적 변화는 `History`와 `Link`에 각각 기록한다.

### 5.1 접근 규칙 (Access Rules)

> [PROHIBIT-1]은 정합성 파괴를 막기 위한 절대 규칙이다. [GUIDELINE-1~3]은 권장 사항이며, 개발자의 판단에 따라 우회할 수 있다. 우회 시 발생하는 정합성 불일치는 `loadstar sync`로 복구하며, 그 책임은 개발자에게 있다.

* **[PROHIBIT-1] `.clionly` 직접 접근 금지** ← 절대 규칙
  AI 에이전트는 `.loadstar/.clionly/` 하위의 어떠한 파일도 직접 읽거나 쓰거나 수정해서는 안 된다. 해당 디렉토리의 모든 읽기·쓰기는 `loadstar` CLI가 전담한다. 직접 접근 시 LOG와 실제 md 상태 간 정합성이 영구적으로 깨진다.

* **[GUIDELINE-1] `.loadstar/` md 직접 편집은 허용하되 sync 필요**
  AI 에이전트는 `.loadstar/` 하위 요소 md 파일(MAP, WAYPOINT 등)을 직접 읽고 수정할 수 있다. 단, 직접 수정 시 메타데이터 LOG가 자동 생성되지 않으므로, 이후 `loadstar sync`를 통해 정합성을 복구해야 한다. 정합성 복구의 책임은 개발자에게 있다.
  - 권장: 가능하면 `loadstar edit [ADDRESS]`를 사용하여 LOG가 자동 생성되도록 한다.
  - 허용: AI가 직접 md를 편집한 경우, 작업 완료 후 사용자에게 sync 필요 여부를 안내한다.

* **[GUIDELINE-2] 메타 동기화 리마인더 활용**
  소스코드 수정 후 관련 WayPoint 갱신 누락 방지를 위해 환경별 메커니즘을 활용한다. Claude Code에서는 PostToolUse Hook이 소스 편집 시 자동 리마인더를 출력하며, AI는 이를 참고하여 관련 WayPoint TODO 체크박스 갱신 여부를 판단한다. Git 커밋은 `git commit`을 직접 사용한다.

* **[GUIDELINE-3] 정합성 불일치 감지 시 사용자에게 안내**
  작업 중 `.clionly`의 LOG와 실제 md 파일 간 불일치가 감지되면, 작업을 계속 진행하되 완료 후 사용자에게 `loadstar sync` 실행을 권고한다.

### 5.2 자원 최적화 및 토큰 관리 규약 (Resource Optimization)
* **지연 로딩 (Lazy Loading)**: `M://` 탐색 시 하위 요소의 세부 사항을 전부 로드하지 않는다. 오직 하위 노드의 주소와 상태 정보만 스캔하여 진입 여부를 결정함으로써 초기 탐색 비용을 최소화한다.
* **실행 시점 접근 (Just-in-Time)**: `W://`의 의도를 분석하는 단계에서는 `S://` 주소의 식별에 그친다. 실제 코드 수정이나 파일 분석이 임박한 실행 직전 단계에서만 해당 물리 리소스의 실제 내용을 컨텍스트에 로드한다.
* **선행 로딩 (Pre-Load)**: 코드 수정 작업 착수 전 진입점 선택은 **작업 유형**에 따라 다음 기준으로 결정한다.

  | 작업 유형 | 진입점 | 이유 |
  | :--- | :--- | :--- |
  | 기능 구현, 설계 변경, 영향 범위 불명확한 수정 | WayPoint → CODE_MAP(scope) → 코드 | CODE_MAP이 탐색 범위를 한정 |
  | 수정 포인트가 명확한 버그 수정·메시지 변경·단일 함수 수정 | grep/직접 탐색 → 코드 → WayPoint 사후 갱신 | WayPoint 경유 비용이 grep보다 크며 맥락 오해 위험이 낮음 |

  - WayPoint 경유 시: CODE_MAP의 scope 범위 내에서 grep/Read로 수정 대상을 특정한다.
  - CODE_MAP이 없는 WayPoint에서 기능 구현 작업이 발생하면 코드를 탐색하여 CODE_MAP을 먼저 작성한 후 작업을 진행한다.
  - SYNCED_AT이 30일 이상 경과한 경우 CODE_MAP의 scope가 여전히 유효한지 확인한 후 작업을 진행한다.
* **완료 후 갱신 (Post-Update)**: 코드 수정 완료 후 WayPoint의 TECH_SPEC 체크박스를 갱신하고 완료 항목에 날짜를 기입한다(`[x] YYYY-MM-DD 항목명`). CODE_MAP scope가 변경되었으면 갱신한다.
* **컨텍스트 슬라이싱 (Context Slicing)**: `H://` 추적 시 전체 이력이 아닌 최신 레코드부터 역순으로 슬라이싱하여 참조한다. 과거의 인과관계 분석이 필수적인 경우에만 상위 경로 주소를 단계적으로 추적하여 결합한다.
* **주소 기반 정밀 타격 (Point Access)**: 광범위한 전체 검색을 지양하고, 모든 데이터 접근은 `요소://` 형식으로 정의된 고유 주소를 통해 대상 파일이나 노드에 즉시 접근하는 핀포인트 방식을 원칙으로 한다.

### 5.3 에이전트 초기화 전략 (Agent Initialization Strategy)

AI 에이전트는 세션이 종료되면 컨텍스트가 초기화된다. 따라서 '학습'은 매 세션 시작 시 자동으로 주입되는 구조로 구현되어야 한다. LOADSTAR는 주입 대상을 성격에 따라 두 가지로 분리하여 관리한다.

**고정 지식 (한 번 주입, 환경에 고정)**
프로젝트가 열리거나 세션이 시작될 때 자동으로 로드되는 불변 규칙이다. LOADSTAR 행동 강령, 주소 체계, 스키마 정의 등이 해당되며, 환경별로 다음 위치에 고정한다.
- **Cursor / IDE**: `.cursor/rules` 또는 `.cursorrules` 파일에 `COMMON/` 핵심 내용의 압축본을 등록한다.
- **대화형 AI (Claude 등)**: System Prompt 또는 Custom Instructions에 고정하여 매 대화마다 자동 주입되도록 한다.
- **MCP 서버**: Tool 호출 방식으로 필요한 시점에 필요한 정보만 반환하며, 별도의 사전 학습이 불필요한 가장 이상적인 형태다.

**변동 상태 (매 세션 로드)**
프로젝트의 현재 상태는 세션마다 달라지므로, 세션 시작 시 단일 진입점 파일(`LOADSTAR_INIT.md`)을 읽어 파악한다.
- `TODO_SPEC.md`: TODO 시스템 규격 (`loadstar todo` 명령 참조)
- 진행 중인 WayPoint의 요약 및 상태

**원칙**: 고정 지식은 환경 설정에 심고, 변동 상태는 `LOADSTAR_INIT.md` 한 파일로 진입한다. 에이전트는 이 두 가지만으로 작업 맥락을 완전히 복원할 수 있어야 한다.
