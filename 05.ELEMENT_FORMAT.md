# 📝 LOADSTAR Unified Element Format

---

## Map vs WayPoint vs Data WayPoint 역할 구분

| 역할 | Map (M://) | WayPoint (W://) | Data WayPoint (D://, dwp) |
| :--- | :--- | :--- | :--- |
| 목적 | WayPoint 탐색을 위한 구조적 길잡이 | 모든 작업의 실행 단위 | 데이터의 개념적 자기소개 |
| STATUS | — (보유하지 않음) | S_IDL / S_PRG / S_STB / S_ERR / S_REV | — (보유하지 않음) |
| WAYPOINTS | 하위 Map / WayPoint / dwp 목록 | — | — |
| CONNECTIONS | — | PARENT, CHILDREN, REFERENCE | PARENT, REFERENCE |
| CODE_MAP | — | 코드 수정 시 탐색 범위 지정 (선택) | 정확한 데이터 스펙 위치 안내 (선택) |
| GOAL | 산하 묶음의 큰 목표 진술 (선택) | 이 작업이 달성하려는 의도 (선택) | — (작업 단위 아님) |
| TODO | — | 실제 작업 체크리스트 (TASK / RECURRING) | — |
| TABLES | — | — | 데이터 본문 — 카테고리/테이블/요소 3단 (선택) |

**원칙**:
- CLI 명령(`loadstar todo`, `loadstar create` 등)의 작업 단위는 항상 WayPoint 기준으로 추적한다. Map은 WayPoint에 도달하기 위한 계층 경로 역할만 한다.
- dwp는 **데이터의 개념적 자기소개**를 담는 노드다. 작업 명세(TODO)는 갖지 않으며, ERD/DDL 수준의 구체 스펙도 담지 않는다(아래 "dwp 작성 원칙" 참조).

## SUMMARY · GOAL · TODO 의미 분담

v1.6.0부터 WP/Map은 세 차원을 의도적으로 분리한다:

| 슬롯 | 답하는 질문 | 비고 |
|:---|:---|:---|
| `SUMMARY` | **이게 무엇인가** (정체성) | 필수. 한 줄로 요소를 식별 |
| `GOAL` | **이게 무엇을 달성하려 하는가** (의도) | 선택. v1.6.0 신설. 한 문장~한 문단 |
| `TODO` | **달성을 위한 구체 작업** | WayPoint에만 존재 |

이 분리의 효용:
- WP 완료 시점에 **TODO가 자기 GOAL을 충실히 실현했는지** 사후 검토 가능.
- Map.GOAL은 산하 WP들이 함께 달성하는 큰 목표를 드러내, 계층 자체가 goal 분해 트리가 된다.
- GOAL이 길어지면 sub-Map/sub-WP로 분해하라는 신호. 한 GOAL이 너무 많은 것을 떠안으면 트리 분해가 약하다는 의미.

## WayPoint 구조 규약

- WayPoint는 하위 WayPoint를 가질 수 있다. 하위 WayPoint는 `CONNECTIONS.CHILDREN`에 `W://` 주소로 등록한다.
- CODE_MAP은 선택 사항이다. 코드 수정이 수반되는 WayPoint에만 작성하며, 순수 계획·조율·문서 작업만 다루는 WayPoint는 생략할 수 있다.
- CODE_MAP이 없는 WayPoint에서 코드 작업이 새로 발생하면 그 시점에 CODE_MAP 섹션을 추가한다.

## WayPoint 포맷

```
<WAYPOINT>
## [ADDRESS] W://경로/ID
## [STATUS] SCHEMA_DEF_기준_코드

### IDENTITY
- SUMMARY: 요소의 목적 및 정체성 요약
- METADATA: [버전, 생성일, 우선순위 등]
- SYNCED_AT: YYYY-MM-DD HH:mm (마지막으로 실제 상태와 동기화한 시각)

### GOAL (선택 — v1.6.0)
이 WayPoint가 달성하려는 의도. 한 문장~한 문단. TODO 항목들이 어떤 목표를 향해 수렴해야 하는지의 기준선.
SUMMARY와 다음과 같이 구분:
- SUMMARY = "JWT 발급 엔드포인트 — 로그인 성공 시 토큰 반환"
- GOAL = "무상태 인증 기반 마련 — SSO 통합과 다중 디바이스 세션을 지원한다"

### CONNECTIONS
- PARENT: 부모 요소 주소 (예: M://..., W://...)
- CHILDREN: 하위 WayPoint 주소 리스트 (예: W://...)
- REFERENCE: 참조 관계 주소 리스트 (예: W://...)

### CODE_MAP (선택 — 코드 수정이 수반되는 경우에만)
- scope: 탐색 범위 디렉토리 경로 (복수 가능)
  - 예: backend/src/main/java/com/loadstar/explorer/service/
  - 예: frontend/src/features/waypoint-editor/
- AI는 scope 범위 내에서만 grep/탐색을 수행하여 수정 대상을 특정한다.

### TODO

WayPoint의 작업 체크리스트. v1.5.0부터 ADDRESS/SUMMARY 래퍼와 TECH_SPEC 컨테이너를 제거하고 TASK / RECURRING 두 헤더 아래 항목을 직접 평탄하게 나열한다.

```
### TODO
# TASK
- [ ] 미완료 항목
- [x] YYYY-MM-DD 완료된 항목

# RECURRING
- (R) 변경 후 mvn test 실행
- (R) ESLint 경고 0개 유지
```

#### TASK 항목 (일회성 구현 작업)
- `- [ ]` 미완료 / `- [x] YYYY-MM-DD` 완료 (완료 날짜 기입)
- AI 재진입 시 체크박스 상태로 "어디서 멈췄는가"를 즉시 파악한다.
- 완료 항목이 과다하게 누적되면 아카이브 파일로 분리할 수 있다. (아래 "TODO 아카이브" 참조)

#### RECURRING 항목 (반복 수행 작업)

**목적**: TASK는 1회성 구현이며 완료되면 `[x]`로 고정된다. 반면 테스트·빌드·린트·포맷팅처럼 **코드가 바뀔 때마다 다시 수행해야 하는 작업**은 별도로 표기하여, AI가 세션마다 자동 재수행 대상을 구분하도록 한다.

**표기**: `- (R) 작업 설명`

**전형적 용례**:
- `(R) 변경 후 mvn test 실행`
- `(R) 변경 후 npm run build로 컴파일 검증`
- `(R) ESLint 경고 0개 유지`
- `(R) API 변경 시 OpenAPI 스펙 동기화`
- `(R) 문자열 변경 시 i18n 키 정합성 검증`

**TASK와의 차이**:

| 속성 | TASK (`[ ]` / `[x]`) | RECURRING (`(R)`) |
|:---|:---|:---|
| 완료 개념 | 있음 (1회 체크로 고정) | 없음 (매번 재수행) |
| S_STB 판정 | TASK 전부 `[x]`면 S_STB | 무관 |
| CLI 표시 (`todo list` 등) | 표시 대상 | 표시 제외 |
| AI 처리 | 착수 전 확인 + 완료 시 체크 | 코드 수정 시점마다 자동 실행 |
| 제거 시점 | 일반적으로 유지 (이력) | 해당 검증이 더 이상 필요 없어진 경우 |

**추가 기준**:
- 해당 WP 산하 코드 변경 시 **반복적으로 수행해야 할 검증·의식**이 발견된 시점에 추가한다.
- 일회성 검증은 TASK로 등록한다(예: `[ ] 초기 DB 마이그레이션 스크립트 실행`).

**제거 기준**:
- 해당 검증이 프로젝트 전역 자동화(pre-commit Hook, CI 파이프라인)로 이관되어 WP 수준 리마인더가 불필요해진 경우.
- 한 WP에 `(R)` 항목이 5개 이상 누적되면, 그 WP의 스코프가 너무 넓거나 CI로 옮겨야 할 징후이므로 재검토한다.

**AI 실행 프로토콜**:
1. TODO의 TASK 항목을 `[x]`로 체크하는 시점에,
2. 같은 WP의 모든 `(R)` 항목을 순차적으로 실행한다.
3. 실패 시 `loadstar log add [WP] ISSUE "..."`로 기록하고 작업을 멈춘다.
4. 성공은 별도 체크하지 않는다(다음 수정 시 다시 실행될 것이므로).

**주의**:
- RECURRING은 **WP 단위** 검증이다. 프로젝트 전역 검증(예: "모든 PR에 테스트 필수")은 CI·Hook·pre-commit에서 강제한다.
- WP 스코프가 좁을수록 `(R)` 항목이 적어지는 것이 정상이다.

### ISSUE
- 설계 시점 알려진 제약/미결 문제
- OPEN_QUESTIONS: 작업 착수 전 사람의 판단이 필요한 미결 결정 사항
  - 전체 상태(OPEN / DEFERRED / CONFIRMED) 정의는 `02.SCHEMA_DEF §4` 참조.
  - 형식: `[Q1] 질문 내용`
  - 확인 완료 시: `[Q1 CONFIRMED <DECISION_ID>] 요약` (Decision 파일 있을 때) 또는 `[Q1 CONFIRMED] 인라인 요약` (단순 Q&A)으로 인플레이스 업데이트. Decision 파일 작성 기준은 아래 Decision 포맷 섹션 참조.
  - AI는 미해결 OPEN_QUESTIONS가 있으면 작업 착수 전 반드시 사람에게 확인을 요청한다.

### COMMENT
- 자유 형식 코멘트
</WAYPOINT>
```

## TODO 아카이브 규칙

WayPoint의 완료 항목이 과다하게 누적되어 토큰 비용이 증가하는 경우, 완료 항목을 별도 파일로 분리할 수 있다.

- **아카이브 파일 위치**: `.loadstar/WAYPOINT/` 디렉토리 내, 원본과 같은 위치
- **명명 규칙**: `원본파일명-archive-NN.md` (NN은 순번)
  - 예: `root.frontend.waypoint_editor-archive-01.md`
- **분리 시점**: S_STB 전환 시 또는 완료 항목이 30개 이상 누적 시
- **아카이브 파일 내용**: 완료된 TODO TASK 항목만 포함 (날짜 포함)
- **WayPoint COMMENT에 참조 기록**: `아카이브: root.frontend.waypoint_editor-archive-01.md`
- **AI 프로토콜**: 아카이브 파일은 일상 작업 시 로드하지 않는다. 과거 이력 조회가 필요할 때만 참조한다.

## Data WayPoint(dwp) 포맷

dwp는 **데이터의 개념적 자기소개**를 담는 1급 노드다. WayPoint가 "작업"이라면 dwp는 "그 작업이 다루는 데이터가 무엇인가"의 정의다.

### dwp 구조 규약

- dwp 본문은 카테고리(SUMMARY) → TABLES(L2) → 요소(L3)의 3단 구조를 권장한다.
- TABLES는 선택이다. 단순한 dwp는 SUMMARY만으로 충분할 수 있다.
- 요소(L3)는 `이름` 또는 `이름: 한 줄 설명`까지만 적는다.
- CODE_MAP은 선택이다. 정확한 스펙(타입/제약/포맷)이 필요하면 그 코드 위치를 안내한다.

### dwp 포맷

```
<DWP>
## [ADDRESS] D://경로/ID

### IDENTITY
- SUMMARY: 이 데이터가 개념적으로 무엇인가 (자기소개)
- METADATA: [Created: YYYY-MM-DD, Updated: YYYY-MM-DD]

### CONNECTIONS
- PARENT: 부모 요소 주소 (예: M://...)
- REFERENCE: 참조 관계 주소 리스트 (예: D://..., W://...)

### TABLES (선택)
- 테이블명1:
  - 요소
  - 요소: 한 줄 설명 (선택)
- 테이블명2:
  - 요소

### CODE_MAP (선택 — 정확한 스펙이 필요한 경우 코드 위치 안내)
- scope: 탐색 범위 디렉토리 경로 (복수 가능)
  - 예: backend/auth/, backend/user/

### ISSUE
- 설계 시점 알려진 제약/미결 문제 (없으면 생략 가능)

### COMMENT
- 자유 형식 코멘트 (없으면 생략 가능)
</DWP>
```

### dwp 작성 원칙

- **추상 우선**: dwp는 데이터의 **개념적 자기소개**다. ERD/DDL 수준의 구체 스펙은 dwp의 영역이 아니다.
- **요소 슬롯 금지 사항**: 타입(string/int 등), 길이/제약(nullable, length=N 등), 인덱스, 외래키 등 **구체 스펙은 적지 않는다**. 정확한 스펙이 필요하면 `CODE_MAP.scope`로 코드를 직접 참조한다.
- **요소 표기**: `이름` 또는 `이름: 한 줄 설명`까지만. 그 이상의 정형 필드는 도입하지 않는다.
- **관계는 작업 측에서 발생한다**: dwp가 다른 dwp와 갖는 관계는 **wp의 작업 기록**이나 **상위 Map의 묶음**에서 자연스럽게 드러나도록 한다. dwp 본문에서 능동적으로 관계 그래프를 그릴 필요 없음(필요 시 `REFERENCE` 슬롯에 주소만 명시).
- **TODO를 갖지 않는다**: dwp는 작업 단위가 아니다. dwp 정의 자체의 변경은 그 dwp를 다루는 wp의 TODO에 항목으로 등록한다.
- **GOAL을 갖지 않는다**: dwp는 의도가 아니라 데이터 자기소개를 담는 노드다. 데이터를 다루는 wp의 GOAL이 그 의도를 진술한다.

## Map 포맷

> **Map은 STATUS를 갖지 않는다.** 진행 상태는 산하 WayPoint의 STATUS가 단일 출처(Single Source)이다. 산하 WP가 모두 S_STB여도 Map의 "완료" 표시는 별도로 두지 않으며, 새 WP가 추가되면 그것이 곧 Map의 "진행 중" 표지다. AI는 Map 상태 동기화 책임을 지지 않는다.

```
<MAP>
## [ADDRESS] M://경로/ID

### IDENTITY
- SUMMARY: Map의 목적 및 구조 요약
- METADATA: [버전, 생성일 등]
- SYNCED_AT: YYYY-MM-DD HH:mm (마지막으로 WAYPOINTS 목록과 실제 구성을 일치시킨 시각)

### GOAL (선택 — v1.6.0)
이 묶음 산하 WayPoint들이 함께 달성하려는 큰 목표. 한 문장~한 문단. 산하 WP들의 개별 GOAL을 묶는 상위 의도 진술.
SUMMARY와 다음과 같이 구분:
- SUMMARY = "인증 모듈 — 로그인/세션/토큰 관리"
- GOAL = "보안 강화와 SSO 지원 — 세션 기반에서 토큰 기반으로 전환하여 다중 클라이언트 호환성을 확보한다"

### WAYPOINTS
- ITEMS: 하위 요소 ID 리스트 (예: M://..., W://..., D://...)

### COMMENT
- 자유 형식 코멘트
</MAP>
```

## Decision 포맷

WayPoint의 OPEN_QUESTIONS가 해결될 때, 결정의 근거·대안·영향을 `.loadstar/DECISIONS/` 하위의 독립 파일로 보존한다. **Architecture Decision Record(ADR) 패턴**을 따른다.

### 언제 Decision 파일을 작성하는가
- OPEN_QUESTIONS가 단순 확인("이 색상 쓸까?") 수준을 넘어 **설계적 판단**을 요구할 때
- 결정이 **여러 WP에 영향**을 주거나, 향후 유사 상황에서 근거를 재참조할 가능성이 있을 때
- 대안 여럿 중 하나를 선택하는 경우 — 탈락 대안의 근거도 남겨두기 위해

단순 Q&A(1~2 문답, 한 WP 내 국소 결정)는 Decision 파일 없이 OPEN_QUESTIONS의 `[Q{N} CONFIRMED]` 라인에 인라인으로만 처리해도 된다.

### 저장 위치 및 명명
- 위치: `.loadstar/DECISIONS/`
- 파일명: `{wp_id}.YYYY-MM-DD.NNN.md` (`04.STORAGE.md §2.2` 참조)
- WP 역참조: `[Q{N} CONFIRMED {wp_id}.YYYY-MM-DD.NNN] 한 줄 요약`

### 포맷

```
<DECISION>
## [ID] {wp_id}.YYYY-MM-DD.NNN
## [STATUS] DECIDED | DEFERRED | SUPERSEDED
## [CREATED_AT] YYYY-MM-DD HH:mm
## [DECIDED_BY] USER | AI | AI_DRAFT_USER_CONFIRMED

### SOURCE
- WP: W://...  (복수 가능: [W://..., W://...])
- Question: [Q{N}] 원본 질문 내용

### CONTEXT
이 질문이 발생한 배경, 관련 제약, 이미 알려진 조건 등.

### OPTIONS
1. **이름** — 설명
   - Pro: ...
   - Con: ...
2. **이름** — 설명
   - Pro: ...
   - Con: ...

### DECISION

### RATIONALE

### IMPACT

```

### 상태 전이

| 상태 | 의미 | 전이 |
|:---|:---|:---|
| **DECIDED** | 결정 완료, 실행 중/완료 | 일반 최종 상태 |
| **DEFERRED** | 당장 결정 불가, 보류 | 후속 DECISION으로 승급 또는 폐기 |
| **SUPERSEDED** | 후속 결정이 덮어씀 | 이전 결정 파일은 보존하고 STATUS만 변경, IMPACT에 후속 파일명 기재 |

### 원칙
- **수정 금지, 추가만**: DECIDED 상태의 결정 파일은 내용을 수정하지 않는다. 번복이 필요하면 새 Decision 파일을 만들고 이전 파일을 SUPERSEDED로 표시한다.
- **Git 이력에 의존**: 결정의 생성/상태 변경은 Git 커밋으로 추적된다.
- **단일 진실**: WP.OPEN_QUESTIONS의 결정 요약(한 줄)은 Decision 파일의 DECISION 섹션과 모순되어선 안 된다. 불일치 시 Decision 파일이 우선.

세션 진입점 파일. AI가 작업 시작 전 프로젝트 컨텍스트를 복원하기 위해 읽는 단일 문서이다.

- **위치**: `.loadstar/LOADSTAR_INIT.md`
- **용도**: AI 세션 시작/종료 시 참고사항을 남기는 프로젝트 레벨 메모
- **작성 주체**: AI 또는 사람이 자유롭게 편집 (UI 대시보드에서 편집 가능)
- **로드 시점**: 세션 시작 시 CLAUDE.md 다음으로 읽는다
