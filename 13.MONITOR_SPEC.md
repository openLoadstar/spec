# 🔍 LOADSTAR Monitor Specification (v0.1)

본 문서는 `.loadstar/` 요소 파일의 실제 변경을 추적하는 모니터링 시스템을 정의합니다.

---

## 1. 설계 목적

### 문제: CLI 변경 기록과 실제 파일 상태의 불일치 가능성

현재 변경 추적 구조:

```
CLI 실행       → .clionly/LOG/  (CLI가 기록, 의도된 변경)
직접 파일 수정 → 추적 수단 없음       ← 공백
```

AI 에이전트는 다음 경우에 변경 기록을 누락할 수 있다:
- 코드를 수정하고 WayPoint TECH_SPEC 체크박스를 갱신하지 않은 경우
- BlackBox CODE_MAP을 수정하지 않고 코드만 변경한 경우
- `loadstar log`를 실행하지 않고 md를 직접 편집한 경우
- 세션이 종료되면서 후처리 작업이 중단된 경우

### 해결 방향

모니터링 프로세스가 `.loadstar/` 요소 파일의 실제 변경 여부를 주기적으로 기록한다.
이 기록은 AI가 직접 읽을 수 있으며, 이후 AI 일관성 검증의 입력 데이터로 사용한다.

> **핵심 원칙**: 모니터링은 변경의 *정확성*을 판단하지 않는다. 오직 *변경 여부*만 기록한다.
> 정확성 판단은 이 데이터를 기반으로 AI가 별도로 수행한다.

## 2. 모니터링 범위

### 추적 대상

`.loadstar/` 하위 요소 파일 중 AI가 접근·수정 가능한 영역:

| 폴더 | 추적 여부 | 이유 |
| :--- | :--- | :--- |
| `WAYPOINT/` | ✅ 추적 | AI 핵심 작업 단위, 갱신 누락 빈번 가능 |
| `BLACKBOX/` | ✅ 추적 | CODE_MAP 드리프트 감지 핵심 |
| `MAP/` | ✅ 추적 | 구조 변경 감지 |
| `.clionly/` | ⬜ 추적 제외 | CLI 전용 영역, 별도 LOG로 관리 |
| `COMMON/` | ⬜ 추적 제외 | 시스템 설정, 빈번한 변경 없음 |

## 3. 저장 구조

### 수명 정책 (Lifecycle)

> **MONITOR/ 는 임시 작업 공간이다. Git 추적 대상에서 제외하며, 조정 세션 완료 후 삭제한다.**

- `.gitignore` 에 `.loadstar/MONITOR/` 등록 필수
- 조정 세션이 끝나면 MONITOR/ 전체를 삭제한다
- 다음 조정 세션 시작 시 `loadstar monitor snapshot` 으로 새로 생성한다
- 스냅샷 간 비교가 목적이므로 장기 보존 데이터가 아니다

이 방식의 이유:
- 조정 세션은 큰 주기로 수행하므로 세션 간 연속성보다 세션 내 정확성이 중요하다
- Git에 MONITOR 데이터가 누적되면 저장소가 불필요하게 커진다
- 조정 완료 후 데이터는 역할을 다한 것이므로 삭제가 자연스럽다

### 위치

```
.loadstar/
├── MONITOR/                       # 임시 작업 공간 (Git 제외, 세션 후 삭제)
│   ├── SNAPSHOT/                  # 조정 세션 중 생성되는 스냅샷
│   │   └── [YYYYMMDD.HHMM].md    # 스냅샷 파일 (시점별)
│   └── DIFF_LOG.md                # 세션 내 스냅샷 간 변경 감지 결과
└── .clionly/
    └── MONITOR/                   # loadstar_monitor 프로세스 전용 (CLI 전용 영역)
        └── checkpoint_needed.flag # 변경 감지 시 생성, checkpoint 후 삭제
```

### 스냅샷 파일 포맷 (`SNAPSHOT/YYYYMMDD.HHMM.md`)

```
## [CAPTURED_AT] 2026-03-26T10:00:00
## [TRIGGER] scheduled | manual | pre-checkpoint

| FILE | MODIFIED_AT | HASH |
| :--- | :--- | :--- |
| WAYPOINT/root.cli.cmd_create.md | 2026-03-26T09:45:00 | a3f1b2c4 |
| WAYPOINT/root.cli.cmd_checkpoint.md | 2026-03-25T14:20:00 | d5e6f7a8 |
| BLACKBOX/root.cli.cmd_create.md | 2026-03-26T09:47:00 | b9c0d1e2 |
| MAP/root.cli.md | 2026-03-24T11:00:00 | f1a2b3c4 |
...
```

- `HASH`: 파일 내용의 짧은 체크섬 (예: SHA1 앞 8자리). 수정 시간이 같아도 내용이 다른 경우 감지.
- `TRIGGER`: 스냅샷이 생성된 원인 (스케줄/수동/checkpoint 직전)

### DIFF_LOG 포맷 (`DIFF_LOG.md`)

연속된 두 스냅샷을 비교하여 변경된 파일 목록을 누적 기록한다.

```
## [DIFF] 20260326.0950 → 20260326.1000
## [CHANGED_FILES]
- WAYPOINT/root.cli.cmd_create.md  | MODIFIED_AT: 09:45 → 09:58 | HASH: a3f1b2c4 → e5f6a7b8
- BLACKBOX/root.cli.cmd_log.md     | NEW (신규 생성)

## [DIFF] 20260326.1000 → 20260326.1010
## [CHANGED_FILES]
- (없음)
```

## 4. 모니터링 트리거

| 트리거 유형 | 시점 | 우선순위 |
| :--- | :--- | :--- |
| **scheduled** | `loadstar_monitor.exe` 프로세스가 5분 간격으로 자동 실행 | ★★★ 최우선 — 변경 감지 시 checkpoint 필요 플래그 생성 |
| **pre-checkpoint** | `loadstar checkpoint` 실행 직전 자동 | ★★ — 코드 커밋 전 메타데이터 상태 포착 |
| **manual** | `loadstar monitor snapshot` 명령 | ★ — 필요 시 수동 |

### 4.1 자동 모니터링 프로세스 (loadstar_monitor.exe)

별도 바이너리 `loadstar_monitor.exe`가 백그라운드에서 상주하며 `.loadstar/` 변경을 감시한다.

* **스캔 주기**: 5분 (기본값)
* **감시 범위**: WAYPOINT/, BLACKBOX/, MAP/ 하위 파일
* **변경 감지 방법**: `git status`로 uncommitted 변경 확인
* **동작**:
    1. 변경 감지 시 `.clionly/MONITOR/checkpoint_needed.flag` 파일 생성 (변경 시각 + 변경 파일 목록 기록)
    2. 이미 flag 파일이 존재하면 중복 생성하지 않음
    3. `loadstar checkpoint` 실행 시 flag 파일 자동 삭제
* **빌드**: `go build -o bin/loadstar_monitor.exe ./cmd/monitor`
* **저장소**: `.loadstar/.clionly/MONITOR/` (CLI 전용 영역)

## 5. CLI 명령

```
loadstar monitor snapshot          # 현재 시점 스냅샷 생성 (MONITOR/ 없으면 자동 생성)
loadstar monitor diff [N]          # 최근 N개 스냅샷 간 변경 목록 출력 (기본값: 2)
loadstar monitor status            # 마지막 스냅샷 이후 변경된 파일 목록 출력
loadstar monitor clean             # 조정 세션 완료 후 MONITOR/ 전체 삭제
```

## 6. AI 일관성 검증 활용 방법

모니터링 데이터는 AI 검증의 *입력*으로 사용된다. AI는 다음 패턴으로 검증을 수행한다.

### 검증 패턴 A: checkpoint 전후 비교

```
pre-checkpoint 스냅샷 → git commit → post-checkpoint 스냅샷
                              ↓
DIFF: 코드 파일은 변경됐는데, 연관 WayPoint/BlackBox 파일은 변경 없음
                              ↓
경고: "cmd_create.go 변경됨, 그러나 WAYPOINT/root.cli.cmd_create.md 미변경"
```

### 검증 패턴 B: 세션 종료 전 점검

```
현재 스냅샷 vs 세션 시작 스냅샷
                ↓
변경된 코드 파일 목록 vs 변경된 메타데이터 파일 목록
                ↓
코드 변경 대비 메타데이터 갱신 누락 항목 리스트업
```

### 검증 패턴 C: SYNCED_AT 드리프트 감지

```
BlackBox의 SYNCED_AT 날짜 이후로 해당 BLACKBOX/*.md 파일이 변경됐는가?
                ↓
변경됐다면: CODE_MAP 갱신 여부 확인 요청
변경 없다면: SYNCED_AT 기준 신뢰 가능
```

## 7. .clionly/LOG/ 와의 관계

| 항목 | `.clionly/LOG/` | `MONITOR/DIFF_LOG` |
| :--- | :--- | :--- |
| 기록 주체 | CLI (명령 실행 시) | 모니터링 프로세스 (주기적) |
| 기록 내용 | 변경 의도 + 필드 전후값 | 파일 수정 여부 + 해시 |
| AI 접근 | 불가 (`.clionly/`) | 가능 (`MONITOR/`) |
| 신뢰 기준 | "무엇을 바꾸려 했는가" | "실제로 무엇이 바뀌었는가" |
| 불일치 시 의미 | CLI가 실행되지 않은 변경 존재 | 예상 변경이 실제로 발생하지 않음 |

두 데이터를 교차 비교하면 더 높은 수준의 정합성 검증이 가능하다.

## 8. 일관성 모델: 허용 범위 내 일관성 (Tolerable Consistency)

### 설계 전제

LOADSTAR는 **완전 일관성(Strong Consistency)을 목표로 하지 않는다.**

코드와 메타데이터의 완전한 동기화가 현실적이지 않은 이유:
- 매 코드 수정마다 WayPoint/BlackBox 전체 갱신은 토큰 비용이 너무 큼
- 사람 중심 개발에서도 문서와 코드의 완전한 일관성 유지는 사실상 불가능
- AI 세션 단절 시 후처리 누락은 구조적으로 발생할 수밖에 없음

따라서 LOADSTAR는 **"드리프트를 알고 있는 상태"** 를 목표로 한다.
모니터링으로 무엇이 얼마나 틀어졌는지를 추적하고, 허용 범위를 초과할 때만 정합성을 맞춘다.

```
[일상 작업]
코드 수정 → 가능한 범위에서 메타데이터 갱신 → 모니터링이 드리프트 기록

[주기적 조정 세션]
DIFF_LOG 분석 → 드리프트가 허용 임계를 초과한 요소만 선택적 갱신
```

### 조정 세션 (Reconciliation Session)

드리프트가 누적되면 별도의 조정 세션을 통해 일괄 정합성을 맞춘다.
조정 세션은 코드 구현 세션과 분리하여 운영한다 (토큰 예산 충돌 방지).

**트리거 조건 (하나 이상 충족 시):**
- DIFF_LOG에 미처리(unresolved) 드리프트 항목이 누적된 경우
- SYNCED_AT 30일 초과 BlackBox가 존재하면서 해당 파일 변경 이력이 있는 경우
- 사용자가 수동 조정 세션 요청

**진행 순서:**
1. `loadstar monitor snapshot` 으로 세션 시작 스냅샷 생성 (MONITOR/ 없으면 자동 생성)
2. `loadstar monitor diff` 로 현재 상태와 이전 스냅샷 간 드리프트 항목 추출
3. 드리프트 항목 중 실제 수정이 필요한 요소 우선순위 분류
4. WayPoint TECH_SPEC 체크박스, BlackBox CODE_MAP 순서로 갱신
5. `loadstar monitor snapshot` 으로 갱신 후 상태 재기록 및 결과 확인
6. 조정 완료 후 `loadstar monitor clean` 으로 MONITOR/ 전체 삭제

**범위 원칙:**
- 한 번의 조정 세션에서 모든 드리프트 해소를 시도하지 않는다
- 현재 ACTIVE WayPoint 관련 요소를 최우선 처리한다
- 조정 불가 항목은 `[DEFERRED]` 마킹 후 다음 세션으로 이월한다

## 9. 미결 설계 사항 (추후 결정)

1. **HASH 알고리즘** — SHA1 앞 8자리 vs xxHash vs 파일 크기+수정시간 조합
2. **post-checkpoint 스냅샷** — checkpoint 완료 후 자동 스냅샷 생성 여부
3. **조정 세션 임계값 수치 정의** — 드리프트 항목 몇 개부터 조정 세션을 권고할지
